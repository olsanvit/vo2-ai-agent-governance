## Prompt Cache Protocol

Při každém startu agent NEčte vždy celý soubor z Drive — nejprve porovná verze:

### Canonical prompt (XxxPrompt.txt + XxxPromptSkills.txt)

1. DB CHECK → SELECT PromptVersion, Content FROM AgentPromptCache
              WHERE AgentType='{AgentType}' AND PromptFile='{PromptFile}'
2. GITEA HEADER → GET https://gitea.vo2info.cz/olsanvit/vo2-ai-agent-governance/raw/branch/main/{PromptFile}
                  (načíst jen první 3 řádky — pouze hlavičku)
3. ROZHODNUTÍ:
   a. cache_version == gitea_version  → použít Content z DB cache; stažení přeskočit
   b. cache_version != gitea_version (nebo cache neexistuje)
      → stáhnout plný obsah z Gitea
      → upsert_record("AgentPromptCache", {AgentType, PromptFile, PromptVersion, Content})
      → použít nový obsah
   c. Gitea nedostupný → zkusit Drive header (fallback)
   d. Drive nedostupný + cache existuje  → použít cache; log "prompt_upgrade_blocked"
   e. Drive nedostupný + cache chybí     → pokračovat s bootstrap verzí; log warning

Stejný protokol platí pro Skills soubor ({PromptFile}Skills.txt).

## Per-agent Files Management

Všechny per-agent soubory žijí na Google Drive. Agent je čte, vytváří a appenduje.

| Soubor | Složka | CREATE | READ | WRITE |
|--------|--------|--------|------|-------|

| {AgentName}_entities.txt | /Prompts/{AgentType}s/ | ✅ při prvním běhu | startup + session cache | ✅ append nové; update RowCount + LastPopulated |

| {AgentName}_urls.txt | /Prompts/Urls/ | ✅ pokud chybí | startup + session cache | ✅ append nové URLs (max 500 řádků) |
| {AgentName}_names.txt | /Prompts/Names/ | ✅ pokud chybí | startup + session cache | ✅ update coverage counts |

CHAPTER 4: COLLECTION WORKFLOW
=============================================================

## Priority Logic

- priority_score = max_count_in_db - current_count_for_category
- Kategorie s nejvyšším priority_score = první na řadě
- Cíl: rovnoměrné pokrytí přes všechny kategorie a CollectionTypes
- Log: "⚖️ Balance: {category} má {count} položek (priority: {score})"

## Collection Protocol

Způsob sběru závisí na konfiguraci CollectionTypes agenta:
- Web search: fulltext dotazy pro každou prioritní kategorii
- Drive scan: prohledávání Drive složek
- API call: volání externího API (dle dostupnosti MCP nástrojů)
- Jiný zdroj: dle specifikace v {AgentName}_config.txt

Rate limiting:
- max MaxPerRun queries/run (default 20)
- min 1s delay mezi požadavky na stejný zdroj
- max 30 req/run per doménu
- Circuit breaker: 3 failures → open 24h

## Classification Rules

Pro každou nalezenou položku:
1. Urči CollectionType dle pravidel z config.txt nebo dle kontextu
2. Zařaď do kategorie z povolených category listů (exact match nebo fuzzy ≥ 80%)
3. Pokud žádná kategorie nesedí → skip + log "⏭️ Skipped: no matching category"

## Idempotency

Před zápisem zkontrolovat business key (definován v entities.txt nebo config.txt).
- Pokud existuje → skip + log "⏭️ Duplicate: {key}"
- Pokud neexistuje → upsert

## Canonical Database Standard

Každá collector tabulka musí obsahovat BaseGuid standard:
- Guid uuid PRIMARY KEY DEFAULT gen_random_uuid()
- CreatedAt timestamptz NOT NULL DEFAULT now()
- UpdatedAt timestamptz NOT NULL DEFAULT now()
- Emoji text NOT NULL DEFAULT ''
- Colors text NOT NULL DEFAULT ''
- + 22 scoring sloupců numeric(6,3)

## Doporučené Collection Columns

Závisí na typu sbíraného obsahu; minimální sada:
- CollectionType text NOT NULL     — typ položky (z CollectionTypes config)
- Category text NOT NULL           — kategorie dle category listu
- SourceUrl text                   — URL zdroje
- ContentUrl text                  — přímá URL obsahu (pokud dostupná)
- Title text                       — název položky
- ConfidenceScore numeric(6,3)     — kvalita/relevance
- IsApproved boolean DEFAULT false — manuální schválení
- IsDeleted boolean DEFAULT false  — soft delete
- Metadata jsonb                   — volitelné extra atributy

Business key: definovaný agentem v entities.txt nebo config.txt.
Typicky: SourceUrl + CollectionType + Category (UNIQUE).

## Quality Filter

Odmítnout pokud:
- Obsah není dostupný (broken link, HTTP != 200)
- Obsah je paywall nebo vyžaduje login
- Obsah je unsafe (explicitní, násilný, nelegální)
- Business key již existuje v DB (duplicate)
- Nesedí do žádné povolené kategorie

ConfidenceScore výpočet (přizpůsobit dle sbíraného obsahu):
- Ověřený zdroj s vysokou spolehlivostí: base 85
- Standardní zdroj: base 70
- Neznámý nebo nízko-spolehlivý zdroj: base 55
- Exact category match: +10
- Přímá URL obsahu dostupná: +5
- Licence ověřena: +5

## Duplicate Detection

Primární: SourceUrl (nebo business key z config)
Sekundární: hash obsahu pokud dostupný

Operátor může pinovat verzi promptu pro konkrétního agenta zápisem do PromptVersionPin (AIData):

- Pin neexistuje nebo PinnedVersion = "latest" → standardní cache protokol
- Prevent: automatické upgrady s breaking changes; agent zůstane na stabilní verzi

## Source Discovery Protocol

Agent aktivně hledá nové zdroje a navrhuje je operátorovi:
1. Každých 10 runů: web search pro nové zdroje dle CollectionTypes
2. Nový zdroj → zapsat do _urls.txt s status=proposed

## Quality Trend Tracking

SourceReliability tabulka rozšířena o trend:
- TrendDirection: improving | stable | declining (porovnání posledních 7 dnů)
- Pokles ReliabilityScore >10 bodů za 7 dnů → ntfy agent-alerts "source declining: {source}"
- Zdroj pod ReliabilityScore 30 → automaticky do circuit breaker open stavu

## Retention Policy

Starší obsah archivován, nikdy hard-deleted:
- IsArchived boolean DEFAULT false
- ArchivedAt timestamptz
- RetentionDays z _config.txt (default 365); záznam starší → IsArchived=true, IsDeleted=false
- Archivace se provede při každém 30. běhu (batch scan)

## Drive fileId Canonical Rule

NIKDY neukládat Google Drive fileId jako canonical identifikátor promptu, skills nebo per-agent souborů.
- Důvod: fileId se změní při re-upload → uložené ID → 404 v dalším runu
- Při každém runu: search_files(query="{filename}", folderId="{canonicalFolderId}") → přesná shoda názvu
- Pokud search vrátí více výsledků: použít POUZE přesnou shodu názvu + nejnovější bezpečně ověřený soubor
- Nejasnost (více souborů se stejným názvem) → reportovat jako blocker nebo degraded dle kritičnosti
- Pokud soubor nenalezen → NIKDY nepoužívat podobný název ani jinou složku

## DB Write Read-Back Rule

Agent nikdy nesmí tvrdit DB zápis bez potvrzeného read-backu.
- Po každém upsert_record nebo smart_upsert_batch: ověřit úspěch z response (rowsAffected > 0)
- Pokud write vrátí 502 nebo chybný response: neclaimovat partial success dokud read-back nepotvrdí
  → run_select_sql("SELECT COUNT(*) FROM {Table} WHERE {criteria}") → ověřit vznik/změnu řádků
- Pokud read-back potvrdí 0 řádků: reportovat no_partial_write_confirmed (ne partial_success)
- Toto pravidlo platí i pro AgentRunReports, AuditLog a problem report soubory

## Memory Non-Canonical Rule

Memory (Claude conversation memory nebo user memory feature) nesmí být canonical source.
- Neukládat canonical prompt, baseline, skills ani provozní stav jako zdroj pravdy do Memory
- Pokud agent uloží Memory poznámku: reportovat ji jako pomocnou kontextovou poznámku, ne jako canonical stav
- Canonical zdroje mají vždy přednost: Gitea → Drive → DB cache → bootstrap (v tomto pořadí)

## Ntfy Runtime Rules

Ntfy posílat pouze přes MCP `send_notification` tool, pokud je skutečně dostupný jako callable tool.
Reportovat jeden z těchto stavů v run reportu:
  ntfy: sent
  ntfy: skipped_capability_missing
  ntfy: failed_403  (chybí NTFY_USER/NTFY_PASS v kontejneru — informovat operátora)
  ntfy: failed_unconfirmed
Ntfy failure je reporting degradation, ne automatický blocker datového běhu,
pokud DB zápisy a validace proběhly bezpečně.

Conflict detection: zakázána. Paralelní běh více agentů je povolený.
