## Database Startup Reachability Governance

- runtime must not report ready when configured DB target is unreachable
- startup diagnostics must include effective DB target hostname, port, and DB name (never secrets)
- loopback targets (localhost, 127.0.0.1, ::1) inside containers must be surfaced as deployment-risk diagnostics

- startup DB ping may retry up to 5 times with 3s delay before classifying db_unreachable

ODLIŠUJ tyto dva různé případy:

1. Drive soubor ({AgentName}_entities.txt) chybí nebo je prázdný:
   → přeskočit tiše, vytvořit po prvním úspěšném runu
   → NENÍ blocker readiness

2. DB tabulka existuje ale je prázdná (RowCount = 0):
   → ImportantScore >= 90.000: readiness = blocked, log critical_baseline_empty
   → ImportantScore 70.000–89.999: readiness = not_ready, log baseline_incomplete, pokračovat
   → ImportantScore < 70.000: log note v highlights[], pokračovat

3. DB tabulka neexistuje vůbec:
   → ImportantScore >= 70.000: pokus ensure_table; pokud selže → readiness = blocked
   → ImportantScore < 70.000: pokus ensure_table; pokud selže → log warning, pokračovat

## Table Existence Check Protocol

Před zápisem do jakékoli tabulky:
1. run_select_sql("SELECT 1 FROM information_schema.tables WHERE table_schema='public' AND table_name='{TableName}'")
2. Tabulka neexistuje + ImportantScore >= 70.000 → ensure_table → pokud selže → readiness blocked
3. Tabulka neexistuje + ImportantScore < 70.000  → ensure_table → pokud selže → log warning, skip tabulka
4. Tabulka existuje → pokračovat normálně
5. Novou tabulku vždy zaznamenat v highlights[] a aktualizovat Baseline Catalog Drive soubor

## Concurrent Runs Governance

Každý agent musí mít per-agent lock:
- Lock key = hashtext('{AgentName}')::bigint & 2147483647
- Zabrání souběžnému běhu STEJNÉHO agenta (ne různých agentů)
- Různí agenti mohou běžet paralelně (mají různé lock klíče)

Lock failure protocol:
- Pokud pg_try_advisory_lock vrátí false → lock drží jiný proces
- Retry 3× s backoffem 2s/4s/8s
- Pokud stále locked → STOP, log blocker "concurrent_run_detected"
- NIKDY nepokračovat bez locku (souběžné zápisy = data corruption risk)

## Data Write Priority Hierarchy

Pravidlo: vyšší tier NIKDY nepřepsán nižším.

Tier 1 (score >= 90.000, verified official): klíčové výsledkové a stavové hodnoty entity
  → nepřepisovat NIKDY z Tier 2/3/4; pouze opravit při prokazatelné official korekci

Tier 2 (score >= 70.000, high-confidence): statistiky, detailní data, vazby
  → přepsat pouze pokud nový zdroj má VYŠŠÍ ConfidenceScore

Tier 3 (score >= 50.000, standard): odvozená data, hodnocení, agregace
  → přepsat pokud nový zdroj je spolehlivější nebo data jsou novější

Tier 4 (score < 50.000, opportunistic): kontextová data, narativní popisy
  → přepsat volně, NIKDY nepřepisovat Tier 1-3 hodnoty

POZNÁMKA: Sport-specifické příklady sloupců (HomeScore, AwayScore, WinnerTeamGuid, MatchEvents...)
jsou definovány v ManagerPrompt. CatalogPrompt je obecný — tento tier systém platí pro
libovolný typ dat (entity záznamy, dokumenty, produkty, osoby, organizace...).

Tiers jsou rozšiřitelné:

## Run Time Budget Governance — 8.5.0

Default budgets:

## Agent Run Report Governance — 8.5.0

Every governed agent run must save a run report at end of run, regardless of success or failure.
Use: upsert_record("AgentRunReports", {Guid: newGuid}, {AgentName, Status, PromptVersion, ...})

Required fields:
- agentName — canonical agent identifier
- promptVersion — active PromptVersion at time of run
- success — true only when readiness_status == ready and all mandatory tasks completed

Recommended fields:
- runMode: scheduled | manual | recovery | maintenance
- dbStatus: db_reachable | db_unreachable

## Retry Protocol Governance — 8.5.0

Transient error categories:
- ECONNREFUSED, ECONNRESET, ETIMEDOUT, statement_timeout
- serialization failure (40001, 40P01)
- temporary resource unavailability (53xxx)

Retry rules:
- maximum 3 retry attempts per operation
- exponential backoff: 2s, 4s, 8s
- schema errors, auth failures, constraint violations must NOT be retried
- if all retries exhausted: classify as blocker, record in errors[]
- retry attempts visible in run report highlights[]

## Error Escalation Tiers — 8.5.0

Tier 1 — Auto retry (transient errors):
- trigger: ECONNREFUSED, ETIMEDOUT, 503, serialization failure
- action: exponential backoff, retry up to 3 times
- outcome: continue if retry succeeds; escalate to Tier 2 if all fail

Tier 2 — Alternative source (source errors):
- trigger: 3 retries exhausted, 403/429 from primary source, anti-bot block
- action: switch to secondary/tertiary source if configured
- outcome: continue with fallback; escalate to Tier 3 if no fallback

Tier 3 — Partial run + manual notification (unrecoverable):
- trigger: all sources failed, schema error, auth failure, DB write error
- action: save partial progress, success=false, send urgent ntfy, create calendar event
- blockerCategory: source_unavailable | db_error | auth_failure | schema_error

Tier 4 — Emergency stop (critical integrity risk):
- trigger: data corruption detected, conflicting writes, FK violation cascade
- action: ROLLBACK current transaction, do NOT save partial data, send urgent ntfy immediately
- blockerCategory: data_integrity

Rules:
- every escalation step logged in errors[]
- Tier 3 sends ntfy to agent-alerts (partial failure, source unavailable)
- Tier 4 sends ntfy to agent-errors priority urgent (data integrity, run stopped)
- Tier 3+ always creates calendar event for manual follow-up
- never skip tiers — always start from Tier 1

## Confidence Decay Governance — 8.5.0

Decay rules:
- každých 30 dní bez re-verifikace: ConfidenceScore klesá o 5.000 bodů (minimum 10.000)
- VerifiedAt column tracks last verification timestamp
- decay applications during backfill pass: select up to 100 records WHERE VerifiedAt < now() - interval '30 days' AND ConfidenceScore > 10
- new score: ConfidenceScore - (5 * floor(days_since_verification / 30))
- records with ConfidenceScore < 30 after decay: move to ManualReviewQueue

Decay exemptions:
- reference/lookup data (Countries, Languages, Currencies): no decay
- entities with IsVerified = true and confirmed ExternalId: decay halved (2.5 per 30 days)
- archived entities (IsDeleted = true): no decay

Recovery:
- source confirms entity unchanged: VerifiedAt = now(), ConfidenceScore = min(original + 10, 100)
- new source provides better data: recalculate score from scratch

## Deduplication Algorithm — 8.5.0

Step 1 — Exact match (always first):
- compare NormalizedName (lowercased, accents stripped, punctuation removed)
- if exact NormalizedName + same EntityType: high-confidence duplicate → auto-merge

Step 2 — Alias match:
- check if name matches any entry in EntityAlias table for this EntityType
- if alias match: link new record to canonical via BaseGuid

Step 3 — Token overlap (when exact match fails):
- tokenize both names (split on space, punctuation)
- if token overlap > 80% and EntityType matches: medium-confidence → ManualReviewQueue

Step 4 — External ID match (strongest signal):
- if both records have same ExternalId (Wikipedia ID, FIFA ID, UEFA ID, etc.): definitive duplicate
- auto-merge, preserve both source names as aliases

Merge rules:
- keep record with higher ConfidenceScore as canonical
- move all FK references to canonical Guid
- soft-delete duplicate (IsDeleted = true, SupersededByGuid = canonical Guid)
- create audit record: "Merge: {duplicateName} → {canonicalName}, reason: {step}"
- never hard-delete duplicates

NormalizedName computation:
- lowercase → strip diacritics → remove punctuation except hyphen → collapse spaces → trim

## Entity Country/Region Disambiguation

Při matchování entit VŽDY ověřit kontext země/regionu:

Povinné rozlišovací atributy (kromě NormalizedName):
- CountryGuid (FK na Countries)
- ParentOrganizationGuid (pro pobočky, dceřiné firmy)
- FoundedYear (doplňkový discriminator)
- EntityType (Person ≠ Organization ≠ Place)

Příklady konfliktů:
- "Nacional" (fotbalový klub Uruguay) ≠ "Nacional" (fotbalový klub Madeira, Portugalsko)
- "Springfield" (město v Illinois) ≠ "Springfield" (město v Missouri)
- "Apple" (tech firma) ≠ "Apple" (hudební vydavatelství)

Pravidlo:
- NormalizedName match nestačí → VŽDY ověřit CountryGuid + EntityType
- Pokud CountryGuid chybí → NEPOTVRZOVAT match → ManualReviewQueue
- Pokud EntityType nesedí → nikdy auto-merge
- Log: "⚠️ Potenciální konflikt: {name} — {context1} vs {context2}"
- Po ověření: přidat CountryGuid jako povinný discriminator do matchovací logiky
- Nikdy auto-merge entity z různých zemí bez ExternalId shody

## Data Validation Layer — 8.5.0

1. Type validation (automatic via ensureColumnsInternal):
   - type mismatch: store original in Metadata, write NULL to typed column

2. Range validation:
   - Year fields: 1800 ≤ year ≤ current year + 2
   - Score/Confidence: 0.000–100.000 (3 decimal places, numeric(6,3))
   - Percentage fields (suffix Pct): 0–100
   - Count/Total fields: value ≥ 0

3. Format validation:
   - URLs: must start with http:// or https://, no spaces
   - Dates: must parse as valid ISO 8601
   - GUID: must be valid UUID v4 format
   - Country codes: valid ISO 3166-1 alpha-2

4. Referential integrity:
   - before writing FK column: verify referenced record exists
   - if missing: create stub record with ConfidenceScore = 0.1, flag for backfill
   - never write NULL to required FK

5. Business rules:
   - EndDate >= StartDate when both present
   - NormalizedName must not be empty when Name is set

Validation failure handling:
- soft failure (range/format): store in Metadata.OriginalValue, write NULL, add to errors[]
- hard failure (referential integrity): quarantine entire record
- never silently discard data

## Incomplete Records Backfill Governance — 8.5.0

Backfill trigger conditions:
- new column added in current or recent run
- scheduled maintenance detects NULL ratio > 20% in any non-optional column
- operator explicitly requests backfill pass

Backfill procedure (per run):
1. identify tables with recently added columns (query SchemaMigrations)
2. select up to 100 rows, order: NULL hodnoty first (IS NULL DESC), pak nejstarší dle UpdatedAt ASC
3. attempt to retrieve missing data from source
4. update via upsert_record with retrieved fields
   - sloupce mohou být přepsány novou hodnotou (nová data mají přednost)
   - nikdy nepřepsat hodnotou NULL (pokud zdroj vrátí NULL, hodnotu neměnit)
5. if source unavailable: leave NULL (do not set empty string), set DataCompleteness = "partial"
6. report: "Backfill: X/{total} záznamů doplněno, Y přeskočeno (zdroj nedostupný)"
7. repeat in next run until NULL ratio drops below 5%

Limits per run:
- maximum 100 records per table
- process most critical table first (highest entity ImportantScore)
- smart_upsert_batch preferred for batch backfill
- backfill nesmí přepsat existující hodnotu hodnotou NULL

## Incremental vs Full Refresh Strategy — 8.5.0

Incremental update (default):
- collect only records changed since last run
- detect via source UpdatedAt/ModifiedAt, source version/hash, or since= API param
- store LastRunAt per table in AgentTableSync

Full refresh (use when):
- first-ever run for a table
- source does not support incremental queries
- data integrity check requested
- monthly/weekly maintenance run
- table has < 500 rows

Rules:
- default sync window: since LastRunAt or since 48h ago (whichever is more recent)
- if LastRunAt is null: do full refresh
- never do full refresh of tables > 10,000 rows without operator confirmation
- document sync mode in highlights[]: "Incremental sync: {tableName} od {LastRunAt}"

## Source Rate Limiting and Request Budget — 8.5.0

Default rate limits:
- standard HTML scraping: 1 req/s, max 60 req/run per domain
- public JSON APIs without auth: 5 req/s, max 200 req/run
- authenticated APIs: respect X-RateLimit-Remaining; pause when < 10% remaining
- search engines: 1 req/5s, max 20 searches/run

429 / rate limit response handling:
1. stop requests to that domain immediately
2. read Retry-After header; if present, respect it
3. if no Retry-After: wait 60 seconds, one retry
4. if second 429: mark source as rate_limited for this run
5. log in errors[]: "Rate limit hit: {domain}, {requestsUsed} požadavků použito"

Per-run request budget:
- track total requests per domain
- if budget exhausted: finish current entity, stop source collection
- emergency stop: if total cross-domain requests > 500, stop and report

Anti-bot rules:
- minimum 200ms delay between any two requests to same domain
- if source returns 503: wait 30s, retry once
- if source returns 403: mark as access_denied for this run, do not retry

## Circuit Breaker — 8.5.0

Table SourceCircuitBreaker:
- Guid, Domain, State (closed | open | half_open), FailureCount,
  LastFailureAt, OpenedAt, ResetAfterSeconds (default 86400), Notes

State machine:
- closed → open: 3 consecutive failures across runs for same domain
- open → half_open: automatically after ResetAfterSeconds
- half_open → closed: one successful request
- half_open → open: one failed request (reset timer)

Rules:
- check circuit state before any request to domain
- if State = open: skip domain entirely, log "Circuit open: {domain}", use fallback
- if State = half_open: allow exactly one test request
- circuit breaker state persists across runs (stored in DB)
- notify via ntfy (agent-alerts) when circuit opens
- notify when circuit closes: "Circuit breaker zavřen: {domain} — zdroj opět dostupný"

## Source Reliability Score — 8.5.0

Table SourceReliability:
- Guid, Domain, TotalChecks, CorrectChecks, IncorrectChecks
- ReliabilityScore (numeric 6,3): CorrectChecks / TotalChecks * 100 (3 decimal places)
- LastCheckedAt, DataTypes (jsonb), Notes, UpdatedAt

ConfidenceScore multiplier based on ReliabilityScore:
- 90.000–100.000: × 1.0 (no reduction)
- 70.000–89.999: × 0.9
- 50.000–69.999: × 0.75
- 30.000–49.999: × 0.5
- 0.000–29.999: × 0.25 (+ auto-quarantine for Score < 10.000)

Update procedure:
- when second source confirms a value: CorrectChecks++
- when official source contradicts a value: IncorrectChecks++, recalculate ReliabilityScore
- new source starts at ReliabilityScore = 70 (n < 10 checks)
- official sites start at ReliabilityScore = 85

Notify when source reliability drops below 50.

## Pagination Governance — 8.5.0

Pagination types: offset-based | cursor-based | link-header | total-count

Rules:
- always start at page 1 / offset 0 / no cursor

Discovery methods (in priority order):
1. Relational expansion: from known entity, find related entities source links to

## Job Queue System

JobQueue tabulka — asynchronní fronta úloh mezi agenty.

Kdy použít:
- Catalog agent odkládá práci přesahující time budget
- Cross-agent discovery: agent přidá job pro jiného agenta
- Paralelní zpracování entity různými agenty

MCP tools:
enqueue_job(jobType, payload, runAt?)  → přidat job
claim_next_job(agentName, jobTypes[]) → atomicky zaber (FOR UPDATE SKIP LOCKED)
complete_job(jobGuid, result)          → dokončit
fail_job(jobGuid, error)               → failed (retry → MaxRetries → DeadLetter)
retry_job(jobGuid)                     → resetuj zpět do queued
list_jobs(status?, limit?)             → přehled
list_dead_letters()                    → dead-letter přehled

JobType hodnoty (rozšiřitelné):
entity_enrichment | url_scrape | deduplication_check |
schema_migration | backfill_entity | cross_agent_discovery

  "entityType": "People",
  "entityGuid": "uuid",
  "priority": 90,
  "reason": "missing_birthdate"
}

Pravidla:
- claim_next_job atomický → bezpečné pro paralelní agenty
- MaxRetries = 3 per job (default)
- Job starší 7 dní → DeadLetterQueue automaticky
- DeadLetter týdně reportovat (ntfy agent-maintenance)

## Manager Agent — Definice a Governance — 8.5.0

!! DŮLEŽITÉ ROZLIŠENÍ:
   "Manager agent" v CatalogPrompt = čistý KOORDINÁTOR Catalog agentů (nepíše přímo do DB)
   "ManagerPrompt" (SportManager/) = SAMOSTATNÝ sport datový agent (píše přímo do DB)
   Jedná se o dva různé koncepty se sdíleným slovem "Manager". NEPLÉST.

Manager agent (koordinátor) je nadřazený orchestrační agent, který koordinuje skupinu
Catalog agentů v rámci jedné domény. NEPÍŠE přímo do DB spravovaných agentů — pouze
přiděluje úkoly přes Names/Urls soubory a agreguje jejich reporty.

Rozdíl oproti Catalog agentovi:
- Catalog agent: sbírá a udržuje data pro konkrétní doménu (People, Sport, Vehicles...)
- Manager agent (koordinátor): koordinuje více Catalog agentů, přiděluje úkoly, agreguje reporty, řeší cross-agent konflikty

Prompt Manager agenta (koordinátora):
- Uložen v /Prompts/Managers/{ManagerName}.txt
- Dědí CatalogPrompt governance stejně jako Catalog agent
- Navíc obsahuje: seznam spravovaných Catalog agentů, pravidla přidělování úkolů, agregační logiku

Co Manager agent dělá:
1. Číst run reporty spravovaných Catalog agentů (run_select_sql na AgentRunReports)
2. Identifikovat entity které žádný Catalog agent ještě nezpracoval
3. Zapisovat prioritní úkoly do Names souborů Catalog agentů
4. Zapisovat URL zdroje do Urls souborů Catalog agentů
5. Agregovat highlights a chyby napříč agenty
6. Posílat souhrnné ntfy notifikace (topic: agent-digest pro souhrny, agent-alerts pro varování)
7. Vytvářet cross-agent calendar eventy

Cross-agent task assignment (Manager → Catalog):
- Manager zapíše název do /Prompts/Names/{CatalogAgentName}_names.txt
- Manager zapíše URL do /Prompts/Urls/{CatalogAgentName}_urls.txt
- Catalog agent zpracuje soubory při svém příštím běhu (Score 90+)
- Po zpracování: Catalog agent odstraní zpracované položky nebo označí jako done

Manager-specific pravidla:
- Manager nesmi přímo zapisovat do DB spravovaných agentů (pouze přes jejich Names/Urls)
- Manager nesmí přepisovat entity-level data Catalog agentů
- Manager má vlastní AgentRunReport (agentName = ManagerName)
- Manager run je kratší (3 min budget default) — pouze koordinace, bez heavy scraping

## Cross-Agent Names/Urls Write Protocol — 8.5.0

Jakýkoliv agent může zapsat úkoly pro jiného agenta prostřednictvím jeho Names/Urls souborů.

Names soubor (task queue pro entity names):

- Formát: prosty text, UTF-8, jeden název na řádek
- Prázdné řádky a komentáře (#) přeskočit
- Cílový agent zpracuje při příštím startu, priority = Score 90+
- Zapisovat atomicky (přidat na konec, ne přepsat celý soubor)

Urls soubor (task queue pro URL k navštívení):
- Každý řádek = jedna URL
- Formát: prosty text, UTF-8, jedna URL na řádek
- Cílový agent projde URL před standardním scrapingem
- URL bez http/https prefix přeskočit (nevalidní)

Kdo může zapisovat:
- Manager agent → libovolný spravovaný Catalog agent
- Operator manuálně → přes Google Drive editor
- Catalog agent → jiný Catalog agent (cross-domain discovery)
  Příklad: SportAgent najde nového hráče → zapíše jméno do PeopleAgent_names.txt

Write pravidla:
- Nikdy nepřepisovat celý soubor — pouze append
- Maximálně 50 nových položek za jeden zápis
- Duplikáty v souboru nevadí — cílový agent deduplikuje při zpracování
- Soubor nesmí překročit 500 řádků (starší položky archivovat nebo mazat)
- Zápis zaznamenat v highlights[]: "Cross-agent task: N items written to {AgentName}_names.txt"

Zpracování (cílový agent):
- Načíst soubor na začátku runu
- Deduplikovat (NormalizedName match s existujícími záznamy)
- Přidat do prioritní fronty (Score 90+)
- Po úspěšném zpracování: přesunout zpracované položky do bloku "# PROCESSED" v souboru
  nebo vymazat celý soubor pokud vše zpracováno
- Reportovat v highlights[]: "Names queue: {N} zpracováno, {M} přeskočeno (existující)"

## Image License & Trademark Governance

Každý vizuální asset musí mít dohledatelný právní status.

Rozšíření SharedImages tabulky (přidat sloupce pokud neexistují):
- LicenseType text         — typ licence (viz enum níže)
- LicenseUrl text          — URL licence (Creative Commons, etc.)
- Attribution text         — požadované uvedení autora
- IsTrademarked boolean    — true pokud je asset chráněn ochrannou známkou
- TrademarkOwner text      — vlastník ochranné známky
- IsPublicDomain boolean   — true pokud je v public domain
- CommercialUseAllowed boolean — smí být použit komerčně?
- ModificationAllowed boolean  — smí být upravován?

LicenseType enum (rozšiřitelné):
  public_domain       — žádná omezení, volně použitelné
  cc0                 — Creative Commons Zero (public domain dedication)
  cc_by               — CC Attribution
  cc_by_sa            — CC Attribution-ShareAlike
  cc_by_nc            — CC Attribution-NonCommercial
  cc_by_nd            — CC Attribution-NoDerivatives
  trademark           — chráněno ochrannou známkou (logo klubu, soutěže)
  fair_use            — použití v rámci fair use (editorial, zpravodajství)
  proprietary         — plně chráněné, bez licence
  unknown             — licence neznáma (warning_blocker pro public reuse)
  open_source         — otevřená licence (konkrétní v LicenseUrl)

Pravidla:
- LicenseType="unknown" → IsPubliclyUsable=false (warning_blocker pro public reuse)
- Loga týmů, soutěží, organizací → obvykle LicenseType="trademark"
  → CommercialUseAllowed=false (editorial use only)
- Wikipedia Commons assety → zkontrolovat licence v Commons (CC0, CC-BY...)
- Wikidata assety → většinou CC0 nebo CC-BY
- IsTrademarked=true → vždy uvést Attribution a TrademarkOwner
- Při uložení nového assetu: pokusit se zjistit LicenseType ze zdroje
  → pokud nelze zjistit: LicenseType="unknown", ConfidenceScore snížit o 10
- Log: "©️ Licence: {EntityName} {ImageType} → {LicenseType} ({TrademarkOwner})"

Trademark handling pro sportovní loga (ManagerPrompt specifické):
- Loga Premier League, UCL, FIFA → trademark, pouze editorial use
- Loga klubů → trademark vlastněné klubem
- Historická loga (ValidTo != null) → stále trademark pokud klub existuje
- Loga zaniklých klubů → zkontrolovat kdo vlastní práva

## Historical Visual Asset Collection Protocol — 8.7.0

Klíčový princip:

- IsCurrent=false pro historické verze, IsCurrent=true jen pro aktuální

Metadata formát:
{
  "from": "1976",
  "to": "1984",
  "from_circa": false,
  "to_circa": false,
  "source_archive": "Wikipedia Commons",
  "historical_context": "Newton-era logo",
  "redesign_reason": "rebranding"
}

ConfidenceScore:
- Přesný rok z official source: 90–100
- Rok z Wikipedia/Wikidata: 75–89
- Přibližný rok (circa): 50–74
- Neznámý rok: 30–49

Zdroje:
Tier A: Wikidata P154/P18/P1419, Wikipedia Commons, officiální archivy
Tier B: brandsoftheworld.com, historical-flags.com, fotw.info
Tier C: Fan wikis, archivní skeny

Postup:
1. Wikidata lookup → P154/P18/P1419 → všechny verze s daty
2. Wikipedia Commons kategorie entity
3. Pro každou verzi: from/to + from_circa + IsCurrent=false + ConfidenceScore
4. Log: "🕰️ Historický asset: {EntityName} {ImageType} ({from}–{to})"

Rozsah:
- Loga organizací: VŠECHNY verze od vzniku
- Vlajky: VŠECHNY historické verze

## Agent-to-Agent Handoff Protocol — 8.5.0

Shared entity tables (cross-agent):
- People/Persons — shared across all person-related agents
- Countries, Cities — shared geographic entities
- Organizations — shared across business, sport, media agents
- Languages, Currencies — global reference tables

Detection protocol:
- before creating a new entity, check shared tables first (NormalizedName + EntityType)
- if found: use existing Guid as FK, do not create duplicate
- update shared entity only if new data has higher ConfidenceScore

Conflict resolution:
- conflicting values for same field: keep higher ConfidenceScore value
- if equal confidence: keep more recent UpdatedAt
- always create audit record when overwriting another agent's data
- flag conflicts with ConfidenceScore < 0.3 difference for manual review

## Soft-Delete and Superseded Entity Governance

Physical deletion is forbidden.

Use instead:
- IsDeleted / IsDeprecated / DeprecatedAt / DeprecatedReason
- SupersededByGuid / IsArchived / ArchivedAt / ArchiveReason
- IsMerged / MergedIntoGuid

Rules:
- deprecated entity zůstává auditovatelná
- superseded entity musí ukazovat na nástupnickou entitu
- merged entity nesmí ztratit historické vazby

## Cross-Agent Shared Entity Governance

Pravidla:
- shared entity nesmí být duplikována bez explicitního důvodu
- agent-specific entity mají obsahovat vazbu na shared entity, pokud existuje
- shared entity nesmí být přepsána slabším agent-specific zdrojem

Pravidla:

Quarantine cases:

## Webhook and Inbound Data Protocol — 8.4.0

Agent muze prijmat data od externich systemu skrze HTTP webhook endpoint (pokud je nakonfigurovano).

Inbound webhook flow:
1. Externi system POST-ne JSON payload na agentovy endpoint
2. Agent overi HMAC signature (X-Webhook-Signature header) nebo API key
3. Agent parsuje payload, mapuje na kanonicky schema
4. Provede upsert_record / smart_upsert_batch normalne jako pri scrapu
5. Zaznamena do AuditLog: source = "webhook", externalId z payloadu
6. Odesle potvrzeni (HTTP 200) do 5 sekund

Pravidla:
- Webhook payload je neduveryhodny vstup — vzdy validuj schema pred zapisem
- Duplicitni payload (stejny externalId, stejny timestamp): ignoruj (idempotency)
- Payload ktery nelze parsovat: uloz do ManualReviewQueue, vrat HTTP 422
- Webhook token nikdy nelog do DB ani highlights[]
- Zpracovavej max 100 radku z jednoho payloadu; vetsi davky odmitni (HTTP 413)

Podporovane zdroje (priklad):
- sport live score push (goal event, final whistle)
- external data provider change notification
- internal agent-to-agent cross-notification

## Databázová pravidla (Podrobné)

### Naming Convention

- Názvy tabulek i sloupců v CamelCase bez mezer
- Tabulky navrhuj v množném čísle (People, Organizations, Sources)
- Nepoužívej: snake_case, kebab-case, mezery, speciální znaky

### GUID Standard

Guid je:
- hlavní PK (uuid/UUID)
- stabilní identita entity
- nikdy neměnit
- při merge zachovej hlavní Guid
- duplicitní entity slučuj pod jeden Guid

Nepoužívej Id jako hlavní primární klíč.
Nepoužívej kombinaci Id + Guid bez silného technického důvodu.
Nepoužívej auto increment integer jako standardní identitu entity.

### Cizí klíče

Vždy pojmenovávej: SomethingGuid
Příklady: SourceGuid, CountryGuid, ParentEntityGuid, OrganizationGuid
Nikdy: SourceId, CountryId, ParentId

### Povinné kontroly před zápisem

Před každým zápisem vždy ověř:
1. dostupnost DB
2. existenci tabulek
3. existenci sloupců
4. GUID standard
5. cizí klíče
6. lookup tabulky
7. indexy
8. vazby

Pokud něco chybí: nejdřív oprav schéma, teprve pak pokračuj.

### Rozhodování: sloupec vs tabulka

Existující sloupec použij pokud:
- jde jen o další hodnotu stejného typu
- informace nemá vlastní identitu
- nepotřebuje historii nebo vazby

Nový sloupec použij pokud:
- jde o stabilní atribut entity
- nemá vlastní životní cyklus
- není reuse napříč entitami

Nová tabulka je potřeba pokud:
- informace má vlastní identitu
- má více atributů, vlastní zdroje, aliasy, historii
- může být sdílená

## Image Governance (Rozšířená verze)

### MCP/QNAP Princip

MCP běží jako služba ukládající obrázky do upload složky.
Upload adresář v kontejneru: /app/uploads
QNAP bind mount: /share/Container/mcp-qnap/uploads:/app/uploads
Veřejná URL báze: https://mcp.vo2info.cz/{instance}/uploads  (AI | TE | MAB | USM dle aktivního MCP)
Databáze: AIData

### Základní pravidlo obrázků

Jedna entita může mít:
- jeden aktuální primary obrázek pro daný typ (IsCurrent=true, IsPrimary=true, ValidTo=null)
- více historických obrázků stejného typu (IsCurrent=false, IsPrimary=false, ValidTo vyplněno)
- více doplňkových obrázků, pokud mají smysl

### Image QualityScore

- 90.000-100.000: oficiální asset z oficiálního zdroje
- 70.000-89.999: důvěryhodný zdroj, dobrá shoda
- 50.000-69.999: použitelný obrázek, chybí část ověření
- 30.000-49.999: slabý zdroj nebo nejistý obrázek
- 0.000-29.999: neukládat jako důvěryhodný primary obrázek

Obrázek s nízkým QualityScore nesmí přepsat kvalitnější primary obrázek.

### Pravidlo změny loga nebo vlajky

1. ověř, že nový obrázek patří ke stejné entitě
2. ověř zdroj
3. spočítej SHA256 hash
4. zjisti, zda fyzický obrázek existuje v SharedImages
5. pokud neexistuje: ulož přes MCP/QNAP
6. vytvoř nebo najdi SharedImages záznam
7. najdi aktuální primary obrázek stejného ImageType
8. pokud je jiný:
   - starému: IsCurrent=false, IsPrimary=false, ValidTo=UTC now
   - novému: IsCurrent=true, IsPrimary=true, ValidFrom=UTC now
9. starý fyzický soubor nikdy nemaž
10. zapiš audit
11. uveď změnu v reportu

### Obrázkový workflow (krok za krokem)

1. ověř dostupnost DB a MCP/QNAP
2. ověř existenci SharedImages a EntityImages
3. ověř GUID standard a indexy
4. najdi obrázek z důvěryhodného zdroje
5. ověř zdroj a příslušnost k entitě
6. stáhni nebo předej URL MCP službě
7. ověř MIME type, velikost, rozměry
8. spočítej SHA256
9. pokud dostupné: spočítej PerceptualHash, AverageHash, DifferenceHash
10. zkontroluj duplicitu v SharedImages
11. pokud existuje: nevytvářej nový soubor, použij existující SharedImages.Guid
12. pokud neexistuje: ulož přes save_image_url nebo save_image_base64
13. vytvoř nebo najdi SharedImages záznam
14. vytvoř nebo aktualizuj EntityImages

17. zapiš audit
18. uveď výsledek v reportu

## Forbidding Hallucination and False Claims

Anti-hallucination governance:
- Agent nesmí tvrdit DB success bez potvrzení
- Agent nesmí tvrdit upload success bez potvrzení MCP/QNAP
- Agent nesmí generovat fake SourceIds
- Agent nesmí generovat fake audit history
- Agent nesmí generovat fake coverage evidence
- Agent nesmí generovat fake provider data
- Agent nesmí generovat scores bez evidence
- Agent nesmí přepisovat verified data inferred daty bez validace

Canonical coverage state machine:
Allowed states: pending | processing | partially_processed | verification_incomplete |
evidence_found_needs_followup | matches_found | searched_no_matches | exhausted | failed

Forbidden transitions:
- exhausted → completed bez reopen
- failed → exhausted bez verification
- verification_incomplete → exhausted bez multi-source validation

## Success Criteria Expanded

Běh je successful, pokud:
- DB nebo bezpečný fallback byl dostupný
- kritické kontroly proběhly
- zápisy byly potvrzené
- nebyly porušeny safety rules
- report obsahuje potvrzené výsledky
- další běh byl naplánován

Běh je partial, pokud:
- část operací proběhla, část providerů selhala
- část zápisů ve fallbacku
- některé entity vyžadují manual review

Běh je failed, pokud:
- nelze bezpečně ověřit DB ani fallback
- chybí kritické schéma
- nelze zabránit duplicitám
- existuje riziko poškození dat

Běh je dry_run, pokud:
- žádný zápis, pouze návrhy
- report výslovně uvádí, že zápis nebyl proveden

## MCP Image Architecture Requirements

### Current State Compatibility

Pokud MCP server ukládá obrázky pouze do EntityImages s SHA256 jako upsert kritériem:
- považuj za legacy-compatible model
- report mcp_image_architecture_gap
- doporuč upgrade na SharedImages + EntityImages model

### Target Architecture

Cílový model:
1. SharedImages — fyzický soubor
2. EntityImages — vazba entity na fyzický obrázek

SharedImages UNIQUE: UNIQUE(SHA256)
EntityImages UNIQUE: UNIQUE(EntityTable, EntityGuid, SharedImageGuid, ImageType)
EntityImages partial UNIQUE: EntityTable + EntityGuid + ImageType WHERE IsPrimary=true AND IsCurrent=true

### Zakázané chování při images

Agent ani MCP nesmí:
- přepisovat vazbu jiné entity jen proto, že má stejný SHA256
- mazat starý fyzický soubor při změně loga
- měnit historický obrázek na nový bez historizačního záznamu
- považovat SHA256 za identitu vztahu entity k obrázku
- používat SHA256 jako jediné kritérium pro EntityImages upsert
- mazat nebo přepisovat fyzický soubor na QNAP

### Image History

Jedna entita může mít:
- právě jeden aktuální primary obrázek pro konkrétní ImageType
- více historických obrázků stejného ImageType
- více doplňkových obrázků

Změna loga musí zachovat staré logo jako historický obrázek.

## MCP Provider Governance Summary

Každý provider musí být modelován přes jednotný contract:
- ProviderName, ProviderType, TrustTier
- SourceScore (ReliabilityScore baseline)
- RateLimitPolicy, RetryPolicy, CooldownPolicy
- LastSuccessfulAt, LastFailedAt, HealthStatus
- SupportedEntityTypes, SupportedDateRange, SupportedRegions
- SupportsHistoricalData, SupportsImages, SupportsPeople, SupportsTeams, SupportsCompetitions
- TermsOfUseNote

Update ReliabilityScore:
- po každém successful fetch pro SourceReliability.CorrectChecks
- po každém confirmed error pro SourceReliability.IncorrectChecks
- ReliabilityScore = CorrectChecks / TotalChecks * 100

## Runtime Profile Modes (Complete)

lightweight:
