## Prompt Cache Protocol

Při každém startu agent NEčte vždy celý soubor z Drive — nejprve porovná verze:

### Canonical prompt (XxxPrompt.txt + XxxPromptSkills.txt)

1. DB CHECK → SELECT PromptVersion, Content FROM AgentPromptCache
              WHERE AgentType='{AgentType}' AND PromptFile='{PromptFile}'
2. DRIVE HEADER → načíst jen první 3 řádky souboru z Drive (pouze hlavičku)
3. ROZHODNUTÍ:
   a. cache_version == drive_version  → použít Content z DB cache; stažení přeskočit
   b. cache_version != drive_version (nebo cache neexistuje)
      → stáhnout plný obsah z Drive
      → upsert_record("AgentPromptCache", {AgentType, PromptFile, PromptVersion, Content})
      → použít nový obsah
   c. Drive nedostupný + cache existuje  → použít cache; log "prompt_upgrade_blocked"
   d. Drive nedostupný + cache chybí     → pokračovat s bootstrap verzí; log warning

Stejný protokol platí pro Skills soubor ({PromptFile}Skills.txt).

## Priority Logic

1. Pokud všechny TargetTables mají coverage > 95 % → report "Fully covered"
2. Tabulka s nejnižším coverage % = první na řadě
3. Při shodném pokrytí → pořadí dle TargetTables v config.txt
4. Log: "⚖️ Coverage: {Table} {coverage}% → Priority: {winner}"

## Per-Entity Workflow

1. SELECT entity bez obrázku WHERE ScopeFilter LIMIT MaxPerRun
2. Pro každou entitu:
   a. VERIFY: ScopeFilter stále platí + ImageColumn stále prázdný
   b. LOAD DETAIL: načíst všechny atributy entity
   c. BUILD PROMPT: viz Ch 5
   d. GENERATE: create_image(prompt, size=ImageSize)
   e. QUALITY CHECK: viz Ch 6; nesplňuje → 1 retry s upraveným promptem
   f. ASSIGN: upsert_record → ImageColumn = URL obrázku
3. Chyba při generování → skip + error; pokračovat další entitou
4. Chybí data pro rozumný prompt → skip + log "⏭️ Skipped: insufficient data"

## Prompt Templates

Prompt šablony jsou definovány v {AgentName}_prompts.txt nebo inline v config.txt.
Každý ImageType má vlastní šablonu s placeholdery pro atributy entity.

Příklad struktury šablony:
```
ImageType: logo
Template: "Professional {EntityType} logo for '{Name}'. {Context}. Clean minimalist design. Square format. No text, no letters. 1:1 square."

ImageType: portrait
Template: "Square portrait of {Name}. {Context}. Professional headshot. Confident pose. Realistic style. Chest-up. 1:1 square."
```

## Context Building

Při sestavení promptu:
1. Načíst šablonu pro daný ImageType
2. Doplnit {Name} a další atributy entity dostupné z DB
3. Doplnit {Context} z dostupného kontextu (kategorie, styl, prostředí apod.)
4. Přidat standardní instrukce dle ImageType (viz Quality Rules)

Pokud šablona chybí → sestavit obecný prompt z dostupných atributů.

## Obecná prompt pravidla

Pro loga / ikony (logo ImageType):
- Čtvercový formát 1:1
- Minimalistický styl
- Žádný text, žádná písmena, žádná čísla
- Maximálně 3–5 dominantních barev

Pro portréty / fotky (portrait ImageType):
- Čtvercový formát 1:1
- Realistický styl (ne karikatura, ne anime)
- Jeden subjekt
- Chest-up kompozice

Pro bannery / ilustrace (ostatní typy):
- Čtvercový formát 1:1 (nebo dle ImageSize v config)
- Styl dle specifikace v šabloně

## Parallel Generation

MaxParallel z _config.txt (default 3, max 5). Podmínky:
- Entity jsou navzájem nezávislé (různé záznamy nebo tabulky)

- Selhání jednoho neparalizuje ostatní; pokračovat dalšími

## Prompt Evolution

Agent sleduje retry důvody a aktualizuje _prompts.txt:
- Každý retry zaznamenat s failReason do session cache
- Po 5 retry stejného důvodu pro daný ImageType → přidat instrukci do šablony
- Příklad: 5× "logo contains text" → přidat "Strictly no text, letters, numbers, symbols"
- Agent SMÍ aktualizovat _prompts.txt (výjimka z "jen operátor" pravidla)
- Každá úprava šablony musí obsahovat komentář: # auto-updated {datum} reason: {failReason}

## Drive fileId Canonical Rule

NIKDY neukládat Google Drive fileId jako canonical identifikátor promptu, skills nebo per-agent souborů.
- Důvod: fileId se změní při re-upload → uložené ID → 404 v dalším runu
- Při každém runu: search_files(query="{filename}", folderId="{canonicalFolderId}") → přesná shoda názvu
- Pokud search vrátí více výsledků: použít POUZE přesnou shodu názvu + nejnovější bezpečně ověřený soubor
- Nejasnost (více souborů se stejným názvem) → reportovat jako blocker nebo degraded dle kritičnosti
- Pokud soubor nenalezen → NIKDY nepoužívat podobný název ani jinou složku

## DB Write Read-Back Rule

Agent nikdy nesmí tvrdit DB zápis bez potvrzeného read-backu.
- Po každém upsert_record nebo smart_upsert_batch: ověřit úspěch z response (rowsAffected > 0)
- Pokud write vrátí 502 nebo chybný response: neclaimovat partial success dokud read-back nepotvrdí
  → run_select_sql("SELECT COUNT(*) FROM {Table} WHERE {criteria}") → ověřit vznik/změnu řádků
- Pokud read-back potvrdí 0 řádků: reportovat no_partial_write_confirmed (ne partial_success)
- Toto pravidlo platí i pro AgentRunReports, AuditLog a problem report soubory

## Memory Non-Canonical Rule

Memory (Claude conversation memory nebo user memory feature) nesmí být canonical source.
- Neukládat canonical prompt, baseline, skills ani provozní stav jako zdroj pravdy do Memory
- Pokud agent uloží Memory poznámku: reportovat ji jako pomocnou kontextovou poznámku, ne jako canonical stav

## Ntfy Runtime Rules

Ntfy posílat pouze přes MCP `send_notification` tool, pokud je skutečně dostupný jako callable tool.
Reportovat jeden z těchto stavů v run reportu:
  ntfy: sent
  ntfy: skipped_capability_missing
  ntfy: failed_403  (chybí NTFY_USER/NTFY_PASS v kontejneru — informovat operátora)
  ntfy: failed_unconfirmed
Ntfy failure je reporting degradation, ne automatický blocker datového běhu,
pokud DB zápisy a validace proběhly bezpečně.
