## Concurrent Runs Governance -- 8.5.0

Lock key = hashtext(COALESCE(NULLIF(SportName,''), 'unknown') || '|' || AgentName)::bigint & 2147483647
  (per sport + per agent — různé sporty mohou běžet paralelně)
  POZNÁMKA: '|' jako separator zabraňuje kolizím (např. "Foot" + "ball" ≠ "Football" + "")
  SportName nesmí být prázdný — pokud sport nebyl nalezen, STOP před lock krokem

Lock failure protocol:
- pg_try_advisory_lock vrátí false → lock drží jiný proces
- Retry 3× s backoffem 2s/4s/8s
- Pokud stále locked → STOP, log blocker "concurrent_run_detected"
- NIKDY nepokračovat bez locku (souběžné zápisy = data corruption risk)

## Run Time Budget Governance -- 8.5.0

Default budgets:
- scheduled (automated): 8 minut hard limit
- manual: 15 minut hard limit
- maintenance: 20 minut hard limit
- recovery: 5 minut hard limit
- Manager coordinating run: 3 minuty hard limit (koordinace, bez heavy scraping)

Time management rules:
1. zaznamenat čas startu runu při první akci
2. kontrolovat elapsed time před každým novým datumem / entity batch
3. pokud > 80% rozpočtu: dokončit aktuální entitu, přejít přímo na reporting
4. pokud > 95% rozpočtu: uložit partial progress, success = false, blockerCategory = "time_budget"
5. partial progress musí být vždy uložen — nikdy neopustit run bez run reportu

Budget warning thresholds:
- 75% elapsed: přidat do highlights[]: "Čas: 75% rozpočtu využito"
- 90% elapsed: prioritizovat pouze ImportantScore >= 70.000 tabulky

- retry attempts viditelné v run report highlights[]

## Error Escalation Tiers -- 8.5.0

Tier 1 — Auto-resolve (no notification):
- missing optional field, minor source timeout, temporary 429

Tier 2 — Log + highlights[] (no ntfy):
- missing recommended field, soft data validation failure
- source returned unexpected format (parseable)

Tier 3 — ntfy agent-alerts + calendar event:
- DB write partial failure, baseline entity missing, source circuit breaker triggered
- all retries exhausted for critical data

Tier 4 — ntfy urgent (agent-errors) + block run:
- db_unreachable, prompt_upgrade_blocked, critical_baseline_empty
- unhandled exception that stops run

Rules:
- každý escalation krok zalogovat v errors[]
- Tier 3 posílá ntfy na agent-alerts
- Tier 4 posílá ntfy na agent-errors (priority urgent)
- Tier 3+ vždy vytváří calendar event pro manuální follow-up
- nikdy přeskočit tier — vždy začínat od Tier 1

## Casovy rozsah

- Agent smi zpracovavat POUZE historicke zapasy
- Horni hranice = vcerejsi datum vcetne (Timezone = Europe/Prague)
  PŘESNÁ DEFINICE: "včerejšek" = datum D-1 v Europe/Prague TZ, celý den 00:00:00–23:59:59 Prague
  V UTC: zapasy kde StartTimeUtc <= YESTERDAY_PRAGUE_DATE 21:59:59 UTC (resp. 22:59:59 v letním čase)
- Dnesni zapasy: StartTimeUtc > konec vcejrejska UTC → PRESKOCIT, skipped_today_or_future
- Budouci zapasy: PRESKOCIT, oznacit jako skipped_today_or_future
- Vcerejsek kontrolovat VZDY
- Vcerejsek NIKDY neoznacovat jako exhausted

## Historicke cilove datumy

- Kazdy napasovany beh zpracuje maximalne 11 cilovych datum
- Prvni cilove datum je VZDY vcerejsek
- Dalsich 10: nejblizsi starsi datumy, ktere NEJSOU ve stavu exhausted
- Datumy ve stavu pending_recheck = znovu aktivni kandidati
- Preferuj datumy kde posledni hledani jeste naslo zapasy
- Preferuj datumy s VerificationStatus = evidence_found_needs_followup
- Preferuj datumy s ProcessingStatus = partially_processed
- MAXIMALNI STARI historickeho data: Sports.MaxHistoricalDaysBack dní
  (default = 1825 dní = 5 let; nastavitelné per sport v Sports tabulce)
  → data starší než MaxHistoricalDaysBack se nezpracovávají, i když jsou not-exhausted
  → SQL filter: TargetDate >= CURRENT_DATE - INTERVAL '{MaxHistoricalDaysBack} days'
  → pokud Sports.MaxHistoricalDaysBack není nastaven: použít default 1825

## Per-date limity

- Max 100 novych ulozenych zapasu na jedno datum
- Max 100 aktualizovanych zapasu na jedno datum
- Max 25 vyhledavacich dotazu na jedno datum
- Max 10 zdroju na jedno datum
- Pokud limit prekrocen -> ProcessingStatus = partially_processed

## Exhausted pravidlo

Datum oznacit jako exhausted AZ PO:
- 5 neuspesnych bezich
- Kazdy pocitany beh mel dostatecnou vicezdrojovou kontrolu:
  1) kontrola oficialniho zdroje (federace, liga, klub)
  2) alespon jeden duveryhodni sportovni vysledkovy web
  3) alespon jeden obecny webovy dotaz (sport + datum + zeme/soutez)
- Minimalne 2 nezavisle zdroje skutecne overeny
- Pokud overeni nebylo dostatecne -> VerificationStatus = verification_incomplete (ne exhausted)
- Pokud existuje naznak existence zapasu -> evidence_found_needs_followup, NoMatchCount NEZVEDAT
- Vcerejsek NIKDY neoznacovat jako exhausted

Automaticke znovuotevreni exhausted datumu:
- Kdyz IsExhausted = true a ExhaustedAt je starsi nez 1 rok -> znovu otevrit
- Nastavit: IsExhausted=false, IsPendingRecheck=true, VerificationStatus=pending_recheck
- Zachovat: historicky NoMatchCount, ExhaustedAt, ExhaustedReason, LastSearchEvidence

## Entity Lifecycle States -- 8.5.0

EntityStatus values:
- draft — nově objevena, ConfidenceScore < 50, zatím neověřena
- active — ověřena, ConfidenceScore >= 50
- stale — active ale data neobnovena v SLA periódě
- archived — již neaktivní (ukončena, uzavřena)
- deleted — soft-deleted, IsDeleted = true
- quarantined — podezřelá data, blokována z reportů
- pending_merge — kandidát na duplikát, čeká na merge rozhodnutí

Transition rules:
- draft → active: ConfidenceScore >= 50 + potvrzení z alespoň 2 zdrojů
- active → stale: SLA překročena
- active → archived: entita ukončena, sezona skončila, hráč se stáhl
- any → quarantined: detekována anomálie nebo integrity conflict
- quarantined → active: manuální schválení operátorem

## Competition Tiebreaker Rules -- 8.5.0

Kazda Competition musi mit konfigurovanana tiebreaker pravidla.
Ulozit v Competition.TiebreakerJson.

Format:
{
  "rules": [
    "points",
    "head_to_head_points",
    "head_to_head_goal_difference",
    "goal_difference",
    "goals_scored",
    "away_goals_scored",
    "disciplinary_points",
    "coin_toss"
  ]
}

Bezte profily:
- UEFA Champions League: points > H2H points > H2H goal diff > H2H away goals > goal diff > goals scored
- Premier League: points > goal diff > goals scored > H2H points > alphabetical
- Serie A: points > H2H points > H2H goal diff > goal diff > goals scored
- Bundesliga: points > goal diff > goals scored > H2H
- La Liga: points > H2H points > H2H goal diff > goal diff > goals scored

Pravidla:
- Aplikovat tiebreaker pri rovnosti Points mezi tymy
- Pokud TiebreakerJson neni nakonfigurovan: pouzit default: points > goal_difference > goals_scored > H2H
- Dokumentovat pouzity tiebreaker v Standings.Notes pro svazan pozice
- coin_toss = last resort, vzdy flag pro manual review

## Coverage Gap Report — 8.5.0

Coverage gap = datum pro dany sport ktere existuje v SportDateCoverage ale nema zadne zaznamy v Matches,
nebo datum ktere chybi uplne (mezera v historii).

Jak identifikovat gaps:
1. Ziskat rozsah historickych dat: MIN(MatchDate) az yesterday ze Matches WHERE SportGuid = X
2. Porovnat s SportDateCoverage tabulkou (ExhaustedAt, MatchCount)
3. Datum kde MatchCount = 0 a ExhaustedAt IS NULL = prioritni gap
4. Datum kde MatchCount = 0 a ExhaustedAt > 1 rok = znovuotevreny gap

Gap prioritizace (Score):
- Datum s hracim kolom (liga round) a MatchCount = 0 → Score 100 (kriticka mezera)
- Playoff / Final datum bez zapasu → Score 95
- Regularni season datum → Score 70
- Preseason datum → Score 30

Coverage Gap Report format (do highlights[]):
  COVERAGE_GAP: SportName | DateRange | GapCount gaps | TopGap: YYYY-MM-DD (Score N)

Coverage Gap ntfy (jednou tydne, jen pokud gaps > 5):
  Title: "Coverage Gap Alert: {SportName}"
  Body: "{N} date gaps found | Oldest: {OldestDate} | Priority: {TopScore}"
  Priority: default

Pravidla:
- Coverage gap report generovat vzdy na konci scheduled runu (po backfill kroku)
- Gaps se Score >= 90 pridat do DiscoveryQueue pro prioritni doplneni
- Pokud celkovy gap count > 50, eskaluj na Tier 2 ntfy notifikaci

## Anti-Duplicate Match Detection — 8.5.0

Stejny zapas muze prichazet z vice zdroju s ruznymi externimi ID.

Detekce duplicit:
Kandidat na duplikat = zapas kde plati VSECHNY:
  - HomeTeamGuid = existujici HomeTeamGuid
  - AwayTeamGuid = existujici AwayTeamGuid
  - ABS(StartTimeUtc - existujici StartTimeUtc) <= 4 hodiny
  - CompetitionGuid = existujici CompetitionGuid (nebo IS NULL)

Algoritmus:
1. Pred insertem noveho zapasu spust: queryRecords(Matches, homeTeam + awayTeam + date + competition)
2. Pokud nalezen kandidat: porovnej ExternalId a SourceName
3. Stejny ExternalId, stejny Source → skip (idempotent)
4. Jiny ExternalId, jiny Source → merge: aktualizuj lepsi dostupna data, uloz oba ExternalId do EntityIdentifiers
5. Jiny ExternalId, stejny Source → log do ManualReviewQueue (neocekavany stav)

Merge pravidla:
- ConfidenceScore = MAX obou zaznamu
- Scores (HomeScore/AwayScore): preferuj zdroj s vyssi SourceReliabilityScore
- Keeper ExternalId: uloz oba do EntityIdentifiers (EntityTable="Matches")
- Pokud uz existuji MatchEvents pro oba → merge events (dedup per PlayerGuid+Minute+EventType)

Log kazdy merge do EntityHistory (ChangeReason = "duplicate_merge").

## Confidence Decay Governance -- 8.5.0

Decay rules:
- každých 30 dní bez re-verifikace: ConfidenceScore klesá o 5.000 bodů (minimum 10.000)
- VerifiedAt column tracks last verification timestamp
- decay applications during backfill pass: select up to 100 records WHERE VerifiedAt < now() - interval '30 days' AND ConfidenceScore > 10
- new score: ConfidenceScore - (5 * floor(days_since_verification / 30))
- records with ConfidenceScore < 30 after decay: move to ManualReviewQueue

Decay exemptions:
- reference/lookup data (Competitions, Sports, Countries): no decay
- entities with IsVerified = true and confirmed ExternalId: decay halved (2.5 per 30 days)
- archived entities (IsDeleted = true): no decay

Recovery:
- source confirms entity unchanged: VerifiedAt = now(), ConfidenceScore = min(original + 10, 100)
- new source provides better data: recalculate score from scratch

## Deduplication Algorithm -- 8.5.0

Step 1 -- Exact match (always first):
- porovnat NormalizedName (lowercase, bez diakritiky, bez interpunkce)
- pokud exact NormalizedName + same EntityType: high-confidence duplicate → auto-merge

Step 2 -- Alias match:
- zkontrolovat, zda name odpovídá záznamu v EntityAlias pro tento EntityType
- pokud alias match: napojit na canonical přes BaseGuid

Step 3 -- Token overlap (when exact match fails):
- tokenizovat oba názvy (split na mezery, interpunkci)
- pokud token overlap > 80% a EntityType souhlasí: medium-confidence → ManualReviewQueue

Step 4 -- External ID match (strongest signal):
- pokud oba záznamy mají stejný ExternalId (FIFA ID, UEFA ID, atd.): definitivní duplikát
- auto-merge, zachovat oba source names jako aliases

Merge rules:
- zachovat záznam s vyšším ConfidenceScore jako canonical
- přesunout všechny FK reference na canonical Guid
- soft-delete duplikát (IsDeleted = true, SupersededByGuid = canonical Guid)
- vytvořit audit záznam v EntityHistory: "Merge: {duplicateName} → {canonicalName}"
- nikdy hard-delete duplikáty

NormalizedName computation:
- lowercase → strip diacritics → remove punctuation except hyphen → collapse spaces → trim

## Data Validation Layer -- 8.5.0

1. Type validation:
   - type mismatch: uložit original do Metadata, zapsat NULL do typed column

2. Range validation:
   - Year fields: 1800 ≤ year ≤ current year + 2
   - Score/Confidence: 0.000–100.000 (3 decimal places, numeric(6,3))
   - Percentage fields (suffix Pct): 0–100
   - Count/Total fields: value ≥ 0

3. Format validation:
   - URLs: musí začínat http:// nebo https://, bez mezer
   - Dates: musí být validní ISO 8601
   - GUID: musí být validní UUID v4 format
   - Country codes: valid ISO 3166-1 alpha-2

4. Referential integrity:
   - před zápisem FK column: ověřit, že referenced záznam existuje
   - pokud chybí: vytvořit stub záznam s ConfidenceScore = 0.1, označit pro backfill
   - nikdy nezapsat NULL do required FK

5. Business rules:
   - EndDate >= StartDate pokud oboje přítomno
   - NormalizedName nesmí být prázdný pokud je Name nastaveno

Validation failure handling:
- soft failure (range/format): uložit do Metadata.OriginalValue, zapsat NULL, přidat do errors[]
- hard failure (referential integrity): quarantine celý záznam
- nikdy tiše zahodit data

## Incomplete Records Backfill Governance -- 8.5.0

Backfill trigger conditions:
- nový sloupec přidán v aktuálním nebo nedávném runu
- scheduled maintenance detekuje NULL ratio > 20% v non-optional sloupci
- operátor explicitně požádá o backfill pass

Backfill procedure (per run):
1. identifikovat tabulky s nedávno přidanými sloupci (query SchemaMigrations)
2. vybrat max 100 řádků, pořadí: NULL hodnoty first (IS NULL DESC), pak nejstarší dle UpdatedAt ASC
3. pokusit se získat chybějící data ze zdroje
4. aktualizovat přes upsert_record s načtenými poli
   - sloupce mohou být přepsány novou hodnotou (nová data mají přednost)
   - nikdy nepřepsat hodnotou NULL (pokud zdroj vrátí NULL, hodnotu neměnit)
5. pokud zdroj nedostupný: ponechat NULL (ne prázdný string), DataCompleteness = "partial"
6. report: "Backfill: X/{total} záznamů doplněno, Y přeskočeno (zdroj nedostupný)"
7. opakovat v příštím runu dokud NULL ratio neklesne pod 5%

Limits per run:
- maximum 100 záznamů na tabulku
- nejprve kritická tabulka (nejvyšší entity ImportantScore)
- smart_upsert_batch preferováno pro batch backfill
- backfill nesmí přepsat existující hodnotu hodnotou NULL

## Incremental vs Full Refresh Strategy -- 8.5.0

Incremental update (default):
- sbírat pouze záznamy změněné od posledního runu
- detekovat přes source UpdatedAt, source version/hash, nebo since= API param
- ukládat LastRunAt per table v AgentTableSync

Full refresh (použít když):
- první run pro tabulku
- zdroj nepodporuje inkrementální dotazy
- data integrity check vyžádán
- monthly/weekly maintenance run
- tabulka má < 500 řádků

Rules:
- default sync window: od LastRunAt nebo od 48h (podle toho co je novější)
- pokud LastRunAt je null: udělat full refresh
- nikdy full refresh tabulek > 10,000 řádků bez potvrzení operátora

- Více infrastruktury = více míst kde může nastat selhání

- Compliance požadavky (GDPR oddělení PII dat)
- Různé backup/retention politiky

## Webhook and Inbound Data Protocol -- 8.5.0

Agent může přijímat data od externích systémů skrze HTTP webhook endpoint (pokud nakonfigurováno).

Inbound webhook flow:
1. Externi system POST-ne JSON payload na agentovy endpoint
2. Agent overi HMAC signature (X-Webhook-Signature header) nebo API key
3. Agent parsuje payload, mapuje na kanonicky schema
4. Provede upsert_record normalne jako pri scrapu
5. Zaznamena do AuditLog: source = "webhook", externalId z payloadu
6. Odesle potvrzeni (HTTP 200) do 5 sekund

Pravidla:
- Webhook payload je neduveryhodny vstup — vzdy validuj schema pred zapisem
- Duplicitni payload (stejny externalId, stejny timestamp): ignoruj (idempotency)
- Payload ktery nelze parsovat: uloz do ManualReviewQueue, vrat HTTP 422
- Webhook token nikdy nelog do DB ani highlights[]
- Zpracovavej max 100 radku z jednoho payloadu; vetsi davky odmitni (HTTP 413)

Doporučený workflow pro SVG:

5. Uložit pouze clean SVG (bez inline scriptů, bez externích referencí)

SVG jsou preferované pro loga a ikony (vektorová kvalita, malá velikost).

## Image License & Trademark Governance

Každý vizuální asset musí mít dohledatelný právní status.

Rozšíření SharedImages tabulky (přidat sloupce pokud neexistují):
- LicenseType text         — typ licence (viz enum níže)
- LicenseUrl text          — URL licence (Creative Commons, etc.)
- Attribution text         — požadované uvedení autora
- IsTrademarked boolean    — true pokud je asset chráněn ochrannou známkou
- TrademarkOwner text      — vlastník ochranné známky
- IsPublicDomain boolean   — true pokud je v public domain
- CommercialUseAllowed boolean — smí být použit komerčně?
- ModificationAllowed boolean  — smí být upravován?

LicenseType enum (rozšiřitelné):
  public_domain       — žádná omezení, volně použitelné
  cc0                 — Creative Commons Zero (public domain dedication)
  cc_by               — CC Attribution
  cc_by_sa            — CC Attribution-ShareAlike
  cc_by_nc            — CC Attribution-NonCommercial
  cc_by_nd            — CC Attribution-NoDerivatives
  trademark           — chráněno ochrannou známkou (logo klubu, soutěže)
  fair_use            — použití v rámci fair use (editorial, zpravodajství)
  proprietary         — plně chráněné, bez licence
  unknown             — licence neznáma (warning_blocker pro public reuse)
  open_source         — otevřená licence (konkrétní v LicenseUrl)

Pravidla:
- LicenseType="unknown" → IsPubliclyUsable=false (warning_blocker pro public reuse)
- Loga týmů, soutěží, organizací → obvykle LicenseType="trademark"
  → CommercialUseAllowed=false (editorial use only)
- Wikipedia Commons assety → zkontrolovat licence v Commons (CC0, CC-BY...)
- Wikidata assety → většinou CC0 nebo CC-BY
- IsTrademarked=true → vždy uvést Attribution a TrademarkOwner
- Při uložení nového assetu: pokusit se zjistit LicenseType ze zdroje
  → pokud nelze zjistit: LicenseType="unknown", ConfidenceScore snížit o 10
- Log: "©️ Licence: {EntityName} {ImageType} → {LicenseType} ({TrademarkOwner})"

Trademark handling pro sportovní loga (ManagerPrompt specifické):
- Loga Premier League, UCL, FIFA → trademark, pouze editorial use
- Loga klubů → trademark vlastněné klubem
- Historická loga (ValidTo != null) → stále trademark pokud klub existuje
- Loga zaniklých klubů → zkontrolovat kdo vlastní práva

## Historical Logo & Name Governance — Teams, Leagues, Seasons

Každý tým, liga nebo soutěž může mít VÍCE historických log a názvů.
Používat EntityImages s IsCurrent/ValidFrom/ValidTo pro celou historii.

Typy změn entity a správný postup:

1. PŘEJMENOVÁNÍ (stejná entita, nový název):
   - NEMĚNIT Guid entity — je to stejná entita
   - Původní Name → uložit do EntityAliases (AliasType="former_name", ValidTo=datum změny)
   - Nový Name → aktualizovat Name + NormalizedName na entitě
   - Původní logo → EntityImages: IsCurrent=false, ValidTo=datum změny
   - Nové logo → EntityImages: nový řádek, IsCurrent=true, ValidFrom=datum změny
   - Log: "🔄 Přejmenování: {starý název} → {nový název} ({datum})"
   - Příklad: "Cardiff City FC" → "Cardiff City" (2012)

2. REFORMACE (nový klub pod novým názvem, právní nástupce):
   - Původní entita: IsDeleted=false, IsArchived=true, ArchivedAt=datum
   - Nová entita: nový Guid, Name, PredecessorGuid → FK na původní entitu
   - Přidat sloupec PredecessorGuid do Teams/Competitions pokud neexistuje
   - Původní loga zůstávají u původní entity (historické)
   - Nová entita začíná bez loga → discovery v příštím runu
   - Log: "🆕 Reformace: {nový název} (nástupce {původní název})"
   - Příklad: Wimbledon FC → AFC Wimbledon (2002)

3. SLOUČENÍ (dva kluby se spojí):
   - Oba původní kluby: IsArchived=true, ArchivedAt=datum sloučení
   - Nová entita: nový Guid, MergedFromGuids (jsonb: [guid1, guid2])
   - Přidat MergedFromGuids (jsonb) do Teams/Competitions
   - Loga obou předchůdců zachovat jako historická v EntityImages (EntityGuid = nová entita)
   - Log: "🤝 Sloučení: {klub1} + {klub2} → {nový název}"
   - Příklad: Gretna FC + Gretna 2008 (různé kluby)

4. SEZÓNNÍ LOGO (liga má různé logo per sezóna):
   - EntityImages.EntityType = "Seasons" (ne "Competitions")
   - ValidFrom = SeasonStartDate, ValidTo = SeasonEndDate
   - IsCurrent = true pouze pro aktuální sezónu
   - Historická sezónní loga uchovávat navždy

Schema rozšíření (přidat pokud sloupce neexistují):
Teams tabulka:
- PredecessorGuid uuid NULL → FK na Teams (pro reformace)
- SuccessorGuid uuid NULL → FK na Teams (pro přejmenování/reformace)
- MergedFromGuids jsonb NULL → seznam Guidů sloučených entit
- IsReformed boolean DEFAULT false
- IsDefunct boolean DEFAULT false → tým přestal existovat
- DefunctDate date NULL
- NameChanges jsonb NULL → [{fromName, toName, changedAt, reason}, ...]

Competitions tabulka: stejná rozšíření jako Teams

Logo discovery pravidla:
- Při každém runu kontrolovat IsCurrent=true EntityImages pro team/league
- Pokud IsCurrent logo je starší 6 měsíců → ověřit zda nezměnilo logo
- Wikipedia + official website = primární zdroje pro logo history
- Historická loga hledat v: Wikipedia Commons, klubové archivy, Wikidata
- ConfidenceScore historického loga: min 60 pokud zdroj uveden

## Historical Visual Asset Collection Protocol — 9.0.0

Cíl: sbírat VŠECHNY historické verze vizuálních assetů (loga, cresty, vlajky, dresy).
Každá verze má: ImageType + ValidFrom + ValidTo + from/to v Metadata.

Klíčový princip:
- NIKDY nezahodit historický asset kvůli neznámému datu
- IsCurrent=false pro všechny historické (ne-aktuální) verze
- IsCurrent=true POUZE pro aktuálně platnou verzi

Metadata formát pro historické assety:
{
  "from": "1955",
  "to": "1974",
  "from_circa": true,
  "to_circa": false,
  "source_archive": "Wikipedia Commons",
  "historical_context": "Pre-European Cup era badge",
  "redesign_reason": "club_rebranding"
}

ConfidenceScore dle jistoty datování:
- Přesný rok z official source: 90–100
- Rok z reliable archive (Wikipedia, Wikidata): 75–89
- Přibližný rok (circa): 50–74
- Neznámý rok: 30–49

Zdroje pro historické assety (v pořadí priority):
Tier A:
  - Wikidata P154 (logo) + P18 (image) + P1419 (coat_of_arms)
    → ukládá VŠECHNY historické verze s daty redesignů
  - Wikipedia Commons — největší archiv s kategoriemi per entita
  - Officiální klubové/instituční archivy
Tier B:
  - sportslogos.net — kompletní timeline sportovních log a dresů
  - brandsoftheworld.com — logo archiv
  - kitsdb.com / historicalkits.co.uk — databáze dresů per sezóna
  - historical-flags.com + fotw.info — historické vlajky
Tier C:
  - Fan wikis, fóra s archivními skeny

Postup historického sběru:
1. Wikidata lookup → P154/P18/P1419 → stáhnout všechny verze s daty
2. Wikipedia Commons kategorie entity → filtrovat dle roku
3. sportslogos.net / kitsdb.com pro sport entity → timeline
4. Pro každou verzi:
   a) Určit from/to z dostupných zdrojů
   b) from_circa=true pokud přibližný rok
   c) IsCurrent=false pro historické verze
   d) ConfidenceScore dle jistoty
   e) Uložit přes save_image_url / save_image_base64
   f) EntityImages: ValidFrom = from rok, ValidTo = to rok
5. Log: "🕰️ Historický asset: {EntityName} {ImageType} ({from}–{to})"

Rozsah:
- Loga: VŠECHNY verze od vzniku entity
- Cresty: VŠECHNY verze od vzniku klubu
- Dresy: per sezóna od ~1970 (kitsdb.com archiv)
- Vlajky: VŠECHNY verze (zejm. státy se změnami vlajky)
- Season loga: per sezóna pro sledované soutěže

Season loga — specifický workflow:
- Každá sezóna soutěže může mít unikátní logo/branding
- EntityType: "Seasons" (ne "Competitions")
- ImageType: season_logo, season_badge, season_artwork, season_trophy_art
- ValidFrom = SeasonStartDate, ValidTo = SeasonEndDate
- IsCurrent = true pouze pro probíhající sezónu
- Sbírat pro: UCL, Europa League, Conference League, národní ligy
- Zdroj: official competition website + Wikipedia Commons

Příklady:
- FC Barcelona: 7+ verzí crests od 1899
- Manchester United: Newton Heath (1878) → postupné redesigny → současné
- Německo: vlajka různé verze s každým státním zřízením
- UCL: season logo/branding se mění každé 3–4 roky

## Cross-Agent Shared Entity Governance -- 8.5.0

Pravidla:
- shared entity nesmí být duplikována bez explicitního důvodu
- agent-specific entity musí obsahovat vazbu na shared entity, pokud existuje

## Canonical Identity Resolution -- 8.5.0

Podporované identity signály:
- canonical name, aliases, multilingual names, transliterations

- alias není samostatná hlavní entita

## Quarantine Governance -- 8.5.0

Quarantine cases:
- corrupted source payload, suspicious entity merge, low-confidence identity
- impossible statistic (HomeScore > 50, MinutesPlayed > 120)

Rules:
- quarantined records musí zachovat source data
- quarantined records se nesmí objevit jako verified data
- quarantine musí obsahovat reason a release condition
- quarantine není deletion

## AgentSchedules Table Schema

```sql
CREATE TABLE IF NOT EXISTS AgentSchedules (
  Guid              uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  AgentName         text NOT NULL UNIQUE,
  AgentType         text NOT NULL,          -- Catalog | Manager
  ScheduledRunTime  time NOT NULL,          -- e.g. 08:30:00 (Europe/Prague)
  Timezone          text NOT NULL DEFAULT 'Europe/Prague',
  IntervalHours     numeric NOT NULL DEFAULT 24,
  NextRunAt         timestamptz,
  LastRunAt         timestamptz,
  IsActive          boolean DEFAULT true,
  Notes             text,
  CreatedAt         timestamptz NOT NULL DEFAULT now(),
  UpdatedAt         timestamptz NOT NULL DEFAULT now()
);
```

Pravidla:
- Business key: AgentName UNIQUE — každý agent má max 1 záznam
- Na konci každého runu: UPSERT (AgentName, ScheduledRunTime, LastRunAt=now(), NextRunAt=now()+IntervalHours)
- Na startu: SELECT all active agents + zobrazit přehled plánů (konflikt detection je zakázána)
- DryRun: UPSERT se neprovede

## Drive fileId Canonical Rule

NIKDY neukládat Google Drive fileId jako canonical identifikátor promptu, skills nebo per-agent souborů.
- Důvod: fileId se změní při re-upload → uložené ID → 404 v dalším runu
- Při každém runu: search_files(query="{filename}", folderId="{canonicalFolderId}") → přesná shoda názvu
- Pokud search vrátí více výsledků: použít POUZE přesnou shodu názvu + nejnovější bezpečně ověřený soubor
- Nejasnost (více souborů se stejným názvem) → reportovat jako blocker nebo degraded dle kritičnosti
- Pokud soubor nenalezen → NIKDY nepoužívat podobný název ani jinou složku

## DB Write Read-Back Rule

Agent nikdy nesmí tvrdit DB zápis bez potvrzeného read-backu.
- Po každém upsert_record nebo smart_upsert_batch: ověřit úspěch z response (rowsAffected > 0)
- Pokud write vrátí 502 nebo chybný response: neclaimovat partial success dokud read-back nepotvrdí
  → run_select_sql("SELECT COUNT(*) FROM {Table} WHERE {criteria}") → ověřit vznik/změnu řádků
- Pokud read-back potvrdí 0 řádků: reportovat no_partial_write_confirmed (ne partial_success)
- Toto pravidlo platí i pro AgentRunReports, AuditLog a problem report soubory

## Memory Non-Canonical Rule

Memory (Claude conversation memory nebo user memory feature) nesmí být canonical source.
- Neukládat canonical prompt, baseline, skills ani provozní stav jako zdroj pravdy do Memory
- Pokud agent uloží Memory poznámku: reportovat ji jako pomocnou kontextovou poznámku, ne jako canonical stav
- Canonical zdroje mají vždy přednost: Gitea → Drive → DB cache → bootstrap (v tomto pořadí)

## Ntfy Runtime Rules

Ntfy posílat pouze přes MCP `send_notification` tool, pokud je skutečně dostupný jako callable tool.
Reportovat jeden z těchto stavů v run reportu:
  ntfy: sent
  ntfy: skipped_capability_missing
  ntfy: failed_403  (chybí NTFY_USER/NTFY_PASS v kontejneru — informovat operátora)
  ntfy: failed_unconfirmed
Ntfy failure je reporting degradation, ne automatický blocker datového běhu,
pokud DB zápisy a validace proběhly bezpečně.
