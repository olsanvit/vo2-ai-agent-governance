## Prompt Cache Protocol

Při každém startu agent NEčte vždy celý soubor z Drive — nejprve porovná verze:

### Canonical prompt (XxxPrompt.txt + XxxPromptSkills.txt)

1. DB CHECK → SELECT PromptVersion, Content FROM AgentPromptCache
              WHERE AgentType='{AgentType}' AND PromptFile='{PromptFile}'
2a. MCP CHECK → GET https://mcp.vo2info.cz/governance/{PromptFile} (timeout=10s)
                  (načíst jen první 3 řádky — pouze hlavičku)
3. ROZHODNUTÍ:
   a. cache_version == mcp_version  → použít Content z DB cache; stažení přeskočit
   b. cache_version != mcp_version (nebo cache neexistuje)
      → stáhnout plný obsah z MCP: GET https://mcp.vo2info.cz/governance/{PromptFile}
      → upsert_record("AgentPromptCache", {AgentType, PromptFile, PromptVersion, Content})
      → write-back Drive: update_file_content({PromptFile}, folderId="1GKqFES4r1zoEBsWjfOD0qs2-Tc08a8xQ", obsah nové verze)
        — pokud Drive write selže: logovat warning, run NEBLOKOVAT (write-back je best-effort)
      → použít nový obsah
      → send_notification(topic="agent-runs", title="🔄 {AgentName} — prompt upgradován", message="{PromptFile}: {old_version} → {new_version}", priority="low")
   c. MCP nedostupný → FALLBACK vo2info.cz:
      GET https://vo2info.cz/governance/{PromptFile}
      fallback 3:
   c2. vo2info.cz nedostupná → FALLBACK GitHub raw:
      GET https://raw.githubusercontent.com/olsanvit/vo2-ai-agent-governance/main/governance/{PromptFile}
   d. GitHub nedostupný → zkusit Drive: read_file_content({PromptFile}, folderId="1GKqFES4r1zoEBsWjfOD0qs2-Tc08a8xQ")
   e. Drive nedostupný + cache existuje  → použít cache; log "prompt_upgrade_blocked"
   f. Vše nedostupné + cache chybí       → pokračovat s bootstrap verzí; log warning

Stejný protokol platí pro Skills soubor ({PromptFile}Skills.txt).

### tab 'entities' — sloupce: ImportantScore|TableName|Description|RowCount|LastPopulated
Agent přidává nové tabulky, aktualizuje RowCount + LastPopulated po každém úspěšném běhu.

### tab 'names' — sloupce: EntityName|coverage|tables|lastProcessed
Čtení při startu pro priority queue; agent appenduje nové entity.

### tab 'urls' — sloupce: URL|SourceTier|ReliabilityScore|EntityType|LastChecked|Status
Čtení při startu pro scraping priority; agent appenduje nové URL (max 500 řádků).

### tab 'error' — sloupce: ProblemScore|Category|Description|Timestamp|StepStatus
Append-only; agent přidává při selhání; max 100 řádků.

### tab 'todo' — sloupce: Status|Priority|Created|Category|Text|Result
Agent čte pending při startu, zpracuje jako instrukce, zapíše Status=done/failed + Result.

### tab 'notes' — sloupce: Topic|Note|CreatedAt
Read-only kontext od operátora; agent nikdy nepíše do notes.

### tab 'config' — sloupce: Key|Value|Description
Per-agent přepisy governance defaults; agent načte při startu.

### TODO TAB PROTOKOL (od verze 10.0.3)
- Agent čte `todo` záložku při každém startupu PŘED spuštěním hlavního runu
- Formát řádku: `YYYY-MM-DD | instrukce | status`
  - status: `pending` (čekat na zpracování) / `done` (hotovo) / `skip` (přeskočit)
- Agent zpracuje všechny řádky se statusem `pending`
- Po zpracování aktualizuje status na `done` nebo `skip` (pokud instrukce neplatí pro tento agent)
- Konflikt s canonical governance: canonical governance má vždy přednost, todo instrukce se zaloguje jako `skip` s důvodem
- Pokud todo záložka neexistuje: pokračovat bez chyby

### CONFIG TAB PROTOKOL (od verze 10.0.3)
Platné override klíče:
- ImportantScore: 0-100 (přepíše default pro tuto instanci agenta)
- SLA_hours: celé číslo (max stale hours)
- MaxEntitiesPerRun: celé číslo (max 500)
- SkipTables: čárkou oddělený seznam tabulek (agent je přeskočí)
- PriorityQueue: čárkou oddělený seznam Guidů (zpracuje nejdříve)
- DryRun: true/false (aktivuje dry run mode)

Formát: `klíč: hodnota` (jeden klíč na řádek)
Neplatný klíč: ignorovat, logovat jako config_unknown_key v Metadata
Priority: config tab PŘEPISUJE canonical governance pro danou instanci

### NOTES TAB PROTOKOL (od verze 10.0.3)
- Read-only kontext od operátora — agent NEZAPISUJE do notes záložky
- Agent čte notes záložku při startupu (jednou, ne při každém runu)
- Formát: volný text, max 10 000 znaků
- Použití: kontext pro interpretaci dat, speciální pokyny pro konkrétní entity, sezónní poznámky
- Pokud notes záložka neexistuje nebo je prázdná: pokračovat bez chyby

## Priority Logic

- priority_score = max_count_in_db - current_count_for_category
- Kategorie s nejvyšším priority_score = první na řadě
- Cíl: rovnoměrné pokrytí přes všechny kategorie a CollectionTypes
- Log: "⚖️ Balance: {category} má {count} položek (priority: {score})"

## Collection Protocol

Způsob sběru závisí na konfiguraci CollectionTypes agenta:
- Web search: fulltext dotazy pro každou prioritní kategorii
- Drive scan: prohledávání Drive složek
- API call: volání externího API (dle dostupnosti MCP nástrojů)
- Jiný zdroj: dle specifikace v {AgentName}_config.txt

Rate limiting:
- max MaxPerRun queries/run (default 20)
- min 1s delay mezi požadavky na stejný zdroj
- max 30 req/run per doménu
- Circuit breaker: 3 failures → open 24h

## Classification Rules

Pro každou nalezenou položku:
1. Urči CollectionType dle pravidel z config.txt nebo dle kontextu
2. Zařaď do kategorie z povolených category listů (exact match nebo fuzzy ≥ 80%)
3. Pokud žádná kategorie nesedí → skip + log "⏭️ Skipped: no matching category"

## Idempotency

Před zápisem zkontrolovat business key (definován v entities.txt nebo config.txt).
- Pokud existuje → skip + log "⏭️ Duplicate: {key}"
- Pokud neexistuje → upsert

## Canonical Database Standard

Každá collector tabulka musí obsahovat BaseGuid standard:
- Guid uuid PRIMARY KEY DEFAULT gen_random_uuid()
- CreatedAt timestamptz NOT NULL DEFAULT now()
- UpdatedAt timestamptz NOT NULL DEFAULT now()
- Emoji text NOT NULL DEFAULT ''
- Colors text NOT NULL DEFAULT ''
- + 22 scoring sloupců numeric(6,3)

## Doporučené Collection Columns

Závisí na typu sbíraného obsahu; minimální sada:
- CollectionType text NOT NULL     — typ položky (z CollectionTypes config)
- Category text NOT NULL           — kategorie dle category listu
- SourceUrl text                   — URL zdroje
- ContentUrl text                  — přímá URL obsahu (pokud dostupná)
- Title text                       — název položky
- ConfidenceScore numeric(6,3)     — kvalita/relevance
- IsApproved boolean DEFAULT false — manuální schválení
- IsDeleted boolean DEFAULT false  — soft delete
- Metadata jsonb                   — volitelné extra atributy

Business key: definovaný agentem v entities.txt nebo config.txt.
Typicky: SourceUrl + CollectionType + Category (UNIQUE).

## Quality Filter

Odmítnout pokud:
- Obsah není dostupný (broken link, HTTP != 200)
- Obsah je paywall nebo vyžaduje login
- Obsah je unsafe (explicitní, násilný, nelegální)
- Business key již existuje v DB (duplicate)
- Nesedí do žádné povolené kategorie

ConfidenceScore výpočet (přizpůsobit dle sbíraného obsahu):
- Ověřený zdroj s vysokou spolehlivostí: base 85
- Standardní zdroj: base 70
- Neznámý nebo nízko-spolehlivý zdroj: base 55
- Exact category match: +10
- Přímá URL obsahu dostupná: +5
- Licence ověřena: +5

## Duplicate Detection

Primární: SourceUrl (nebo business key z config)
Sekundární: hash obsahu pokud dostupný

## Drive fileId Canonical Rule

NIKDY neukládat Google Drive fileId jako canonical identifikátor promptu, skills nebo per-agent souborů.
- Důvod: fileId se změní při re-upload → uložené ID → 404 v dalším runu
- Při každém runu: search_files(query="{AgentName}", folderId="{canonicalFolderId}") → přesná shoda názvu master spreadsheet
- Pokud search vrátí více výsledků: použít POUZE přesnou shodu názvu + nejnovější bezpečně ověřený soubor
- Nejasnost (více souborů se stejným názvem) → reportovat jako blocker nebo degraded dle kritičnosti
- Pokud soubor nenalezen → NIKDY nepoužívat podobný název ani jinou složku

## DB Write Read-Back Rule

Agent nikdy nesmí tvrdit DB zápis bez potvrzeného read-backu.
- Po každém upsert_record nebo smart_upsert_batch: ověřit úspěch z response (rowsAffected > 0)
- Pokud write vrátí 502 nebo chybný response: neclaimovat partial success dokud read-back nepotvrdí
  → run_select_sql("SELECT COUNT(*) FROM {Table} WHERE {criteria}") → ověřit vznik/změnu řádků
- Pokud read-back potvrdí 0 řádků: reportovat no_partial_write_confirmed (ne partial_success)
- Toto pravidlo platí i pro AgentRunReports, AuditLog a problem report záznamy v tab 'error'

## Memory Non-Canonical Rule

Memory (Claude conversation memory nebo user memory feature) nesmí být canonical source.
- Neukládat canonical prompt, baseline, skills ani provozní stav jako zdroj pravdy do Memory
- Pokud agent uloží Memory poznámku: reportovat ji jako pomocnou kontextovou poznámku, ne jako canonical stav
- Canonical zdroje mají vždy přednost: Gitea → Drive → DB cache → bootstrap (v tomto pořadí)

## Ntfy Runtime Rules

Ntfy posílat pouze přes MCP `send_notification` tool, pokud je skutečně dostupný jako callable tool.
Reportovat jeden z těchto stavů v run reportu:
  ntfy: sent
  ntfy: skipped_capability_missing
  ntfy: failed_403  (chybí NTFY_USER/NTFY_PASS v kontejneru — informovat operátora)
  ntfy: failed_unconfirmed
Ntfy failure je reporting degradation, ne automatický blocker datového běhu,

## Dynamic Frequency Adjustment -- 10.1.1

Agent smí autonomně upravit svůj interval běhu na základě objemu dat v posledním runu.
Úprava se provádí v kroku 8a (SCHEDULE UPDATE) — pouze pokud AgentSchedules.AllowAutoAdjust = true.

### Povolený rozsah intervalu

| Limit | Hodnota |
|-------|---------|
| Minimum (nejčastěji) | 24 hodin (1× denně) |
| Maximum (nejméně) | 168 hodin (1× týdně) |
| Maximální změna za jeden run | ±50 % aktuálního intervalu |

### Podmínky pro zkrácení intervalu (běžet častěji)

Splnění ALESPOŇ JEDNÉ podmínky → zkrátit interval × 0.75, min 24h:
- Nových záznamů insertováno > 200 v tomto runu
- Nových URLs přidáno do fronty > 30
- Počet aktivních zdrojů s novými daty > 10

### Podmínky pro prodloužení intervalu (běžet méně často)

Splnění ALESPOŇ JEDNÉ podmínky → prodloužit interval × 1.5, max 168h:
- Nových záznamů = 0 ve 3 po sobě jdoucích runech (ConsecutiveEmptyRuns >= 3)
- Celkový počet nových záznamů < 10 v tomto runu
- Všechny aktivní zdroje vrátily 304 Not Modified nebo prázdnou odpověď

### Priorita pravidel

1. IsManualOverride = true v AgentSchedules → VŽDY zachovat operátorem nastavený interval, žádná úprava
2. AllowAutoAdjust = false → žádná úprava (default)
3. DryRun = true → vypočítat nový interval, ale NEULOŽIT do DB
4. Oba pravidla (zkrátit + prodloužit) platí → použít zkrácení (priorita na více dat)

### Postup při úpravě (krok 8a)

1. Načíst aktuální IntervalHours z AgentSchedules
2. Vyhodnotit podmínky na základě run statistik
3. Vypočítat newInterval = clamp(vypočítaný, 24, 168)
4. Pokud newInterval != currentInterval:
   a) upsert_record("AgentSchedules", {AgentName}, {IntervalHours: newInterval, LastAdjustedAt: now(), AdjustmentReason: "<důvod>"})
   b) Logovat do highlights[]: "📅 Interval upraven: {currentInterval}h → {newInterval}h | Důvod: {důvod}"
   c) send_notification(topic="agent-maintenance", title="📅 {AgentName} — interval upraven", message="{currentInterval}h → {newInterval}h | {důvod}", priority="min", tags=["calendar"])
5. Pokud newInterval == currentInterval: žádná akce, žádná notifikace

### AgentSchedules sloupce pro auto-adjustment

| Sloupec | Typ | Popis |
|---------|-----|-------|
| AllowAutoAdjust | boolean | Povolení auto-úpravy (default false) |
| IsManualOverride | boolean | Ochrana před auto-úpravou (operátor nastavil ručně) |
| LastAdjustedAt | timestamptz | Čas poslední auto-úpravy |
| AdjustmentReason | text | Důvod poslední změny intervalu |
