## Prompt Cache Protocol

Při každém startu agent NEčte vždy celý soubor z Drive — nejprve porovná verze:

### Canonical prompt (XxxPrompt.txt + XxxPromptSkills.txt)

1. DB CHECK → SELECT PromptVersion, Content FROM AgentPromptCache
              WHERE AgentType='{AgentType}' AND PromptFile='{PromptFile}'
2. GITEA HEADER → GET https://gitea.vo2info.cz/olsanvit/vo2-ai-agent-governance/raw/branch/main/{PromptFile}
                  (načíst jen první 3 řádky — pouze hlavičku)
3. ROZHODNUTÍ:
   a. cache_version == gitea_version  → použít Content z DB cache; stažení přeskočit
   b. cache_version != gitea_version (nebo cache neexistuje)
      → stáhnout plný obsah z Gitea
      → upsert_record("AgentPromptCache", {AgentType, PromptFile, PromptVersion, Content})
      → použít nový obsah
   c. Gitea nedostupný → zkusit Drive header (fallback)
   d. Drive nedostupný + cache existuje  → použít cache; log "prompt_upgrade_blocked"
   e. Drive nedostupný + cache chybí     → pokračovat s bootstrap verzí; log warning

Stejný protokol platí pro Skills soubor ({PromptFile}Skills.txt).

## Per-agent Files Management

Všechny per-agent soubory žijí na Google Drive. Agent je čte, vytváří a appenduje.

|--------|--------|--------|------|-------|

## SubfolderScheme logika

| Hodnota    | Chování |
|------------|---------|
| none       | Zpracovávat přímo soubory v SourceFolder; bez SubfolderKey |
| by_season  | Podsložky jsou čísla sezón; SubfolderKey = season |
| by_date    | Podsložky jsou data (YYYY-MM nebo YYYY-MM-DD); SubfolderKey = date |
| custom     | Logika definována v config.txt nebo mapping.txt |

SubfolderKey hodnotu vždy určuj z názvu podsložky; nikdy neodhaduj z obsahu souboru.
Pokud SubfolderKey nelze bezpečně určit → označit jako neúplný import; vyžádat upřesnění.

## Pořadí zpracování

- Zpracovávat od nejstarší podsložky k nejnovější (chronologicky nebo dle SubfolderScheme)
- Při catch-up: zpracovat vše dostupné; neomezovat dávku
- Testovací složky (dle config.txt TestFolders): ignorovat v naplánovaném běhu

## Typy souborů

Typy screenshotů / obrázků jsou definovány v {AgentName}_mapping.txt.
Každý typ má svoji sadu extrahovaných polí a pravidla mapování do DB.

Obecná klasifikace:

3. Načíst příslušná mapovací pravidla z mapping.txt

## Skupinové zpracování (Block Anchoring)

Pokud mapping.txt definuje "anchor" typ (první soubor skupiny):
- Z anchor souboru ulož sdílený kontext (např. název entity, SubfolderKey hodnotu)
- Navazující soubory (stejná složka, podobný čas) mohou zdědit kontext z anchor
- Dědění ukonč při změně typu nebo novém anchor souboru
- Nikdy nepropojuj jiný typ souboru s anchorem automaticky

## Pravidla kvality dat

- Pokud je hodnota nečitelná nebo nejednoznačná → null; nezapisovat odhad
- Pokud SubfolderKey nelze bezpečně určit → označit jako neúplný import
- Zkrácené nebo nejasné identifikátory → nezapisovat jako plný záznam bez potvrzení
- Neviditelné nebo domýšlené části → přeskočit

## Data Lineage

Každý záznam musí mít sloupce pro sledování původu dat (rozšířený BaseGuid standard 9.0.0):

| Sloupec | Typ | Výchozí | Pravidlo |
|---------|-----|---------|---------|
| CreatedByAgent | text | '' | Nastavit při insertu; nikdy neměnit |
| LastUpdatedByAgent | text | '' | Aktualizovat při každém UPDATE |
| DataSource | text | '' | Původní zdroj (URL, cesta k souboru, název screenshotu) |

Agent VŽDY nastavuje tyto tři sloupce. Umožňuje zpětně dohledat, kdo co zapsal.

## Environment Support

Agent čte prostředí z {AgentName}_config.txt (pole Environment: prod | dev | staging).

| Environment | DB | Drive | Notifikace | Calendar |
|-------------|----|----- |-----------|---------|
| prod | TargetDB | /Prompts/{AgentType}s/ | agent-runs / agent-errors | ✅ |
| dev | TargetDB nebo dev prefix | /Prompts/Dev/{AgentType}s/ | agent-dev | ❌ |
| staging | staging DB | /Prompts/Staging/{AgentType}s/ | agent-staging | ❌ |

Dev prostředí: všechny DB zápisy jsou automaticky DryRun.
Bootstrap hodnota je výchozí pokud _config.txt neexistuje nebo pole chybí: prod.

## Agent Health Reporting

Na konci každého běhu agent upsertuje záznam do AgentHealthReport (AIData):

```
AgentHealthReport: AgentName, AgentType, Status (healthy/degraded/error),
  PromptVersion, Environment, DBStatus, LastRunAt, LastRunSuccess,
  RunDurationMs, ErrorCount, Metadata jsonb
```

Agent vytvoří tabulku pokud neexistuje. Slouží jako centrální dashboard zdraví agentů.

## Prompt Version Pinning

Operátor může pinovat verzi promptu pro konkrétního agenta zápisem do PromptVersionPin (AIData):

```
PromptVersionPin: AgentType, PinnedVersion ("9.0.0" nebo "latest"), PinnedBy, Reason
```

Agent na startu zkontroluje pin PŘED Prompt Cache Protocol (viz Ch 3):
- Pin existuje a PinnedVersion != "latest" → načíst pinnutou verzi z cache nebo Drive archivu
- Pin neexistuje nebo PinnedVersion = "latest" → standardní cache protokol
- Prevent: automatické upgrady s breaking changes; agent zůstane na stabilní verzi

## Agent Deprecation Lifecycle

Operátor nastaví v {AgentName}_config.txt:

| status | Chování |
|--------|---------|
| active (default) | Normální běh |
| deprecated | Agent běží, loguje každý run jako deprecated; ntfy agent-alerts |
| disabled | Agent okamžitě ukončí startup s "Agent disabled by operator" |

Při archivaci: Drive soubory přesunout do /Archive/{AgentName}/.
Data v DB nikdy nemazat.


## OCR Confidence Scoring

Pro každé vytěžené pole ukládat ConfidenceScore (0–100):
- 90–100: jasně čitelné | 70–89: pravděpodobně správné | 50–69: nejisté | <50: přeskočit
- DB sloupec: {fieldName}Confidence numeric(6,3) (přidávat kde má smysl)
- Pole s ConfidenceScore < 70 → označit v auditu jako "low_confidence"
- Agregátní ImportConfidence = průměr confidence všech uložených polí per screenshot

## Historical Comparison

Při importu nových dat porovnat s předchozím záznamem dle business key:
- Identifikovat výrazné změny (>20% odchylka číselných hodnot)
- Zaznamenat do auditu: "value changed: {field} {old} → {new}"
- Změny >50% označit jako manual_review
- NIKDY nepřepisovat historická data; ukládat nová do nového záznamu s odkazem na původní

## Parallel Parsing

MaxParallel z _config.txt (default 3). Podmínky:
- Nezávislé soubory (různé entity, různé složky) → paralelní parsování
- Závislé soubory (team block) → vždy sekvenční
- Sdílené zdroje (DB schema cache, config) načíst jednou před paralelizací
- DB zápis vždy sekvenční (zabraňuje race conditions)

## Import Rollback

RunGuid (uuid) přiřazen každému běhu. Pokud DB zápis selže:
1. UPDATE TargetTable SET ImportStatus="rolled_back", IsDeleted=true, UpdatedAt=now() WHERE RunGuid = {currentRunGuid} AND ImportStatus = "pending"
2. Auditní záznamy přepsat na status = "rolled_back"
3. Soubory NEPŘESOUVAT — zůstanou ve zdrojové složce
4. READ-BACK CONFIRM: SELECT COUNT(*) FROM TargetTable WHERE RunGuid = {currentRunGuid} AND ImportStatus = "rolled_back" → ověřit shodu s N
5. Reportovat: "Rollback: {N} záznamů označeno jako rolled_back (IsDeleted=true)"
   NIKDY použít DELETE — záznamy zůstávají auditovatelné v DB

## Screenshot Deduplication

ScreenshotImportLog (TargetDB): FileHash, OriginalPath, ImportedAt, RunGuid.
Před parsováním:
1. Spočítat SHA256 hash obsahu souboru
2. SELECT 1 FROM ScreenshotImportLog WHERE FileHash = {hash}
3. Existuje → skip; "⏭️ Duplicate: {filename} (hash match)"
4. Neexistuje → zpracovat; uložit hash po úspěšném importu

## Drive fileId Canonical Rule

NIKDY neukládat Google Drive fileId jako canonical identifikátor promptu, skills nebo per-agent souborů.
- Důvod: fileId se změní při re-upload → uložené ID → 404 v dalším runu
- Při každém runu: search_files(query="{filename}", folderId="{canonicalFolderId}") → přesná shoda názvu
- Pokud search vrátí více výsledků: použít POUZE přesnou shodu názvu + nejnovější bezpečně ověřený soubor
- Nejasnost (více souborů se stejným názvem) → reportovat jako blocker nebo degraded dle kritičnosti
- Pokud soubor nenalezen → NIKDY nepoužívat podobný název ani jinou složku

## DB Write Read-Back Rule

Agent nikdy nesmí tvrdit DB zápis bez potvrzeného read-backu.
- Po každém upsert_record nebo smart_upsert_batch: ověřit úspěch z response (rowsAffected > 0)
- Pokud write vrátí 502 nebo chybný response: neclaimovat partial success dokud read-back nepotvrdí
  → run_select_sql("SELECT COUNT(*) FROM {Table} WHERE {criteria}") → ověřit vznik/změnu řádků
- Pokud read-back potvrdí 0 řádků: reportovat no_partial_write_confirmed (ne partial_success)
- Toto pravidlo platí i pro AgentRunReports, AuditLog a problem report soubory

## Memory Non-Canonical Rule

Memory (Claude conversation memory nebo user memory feature) nesmí být canonical source.
- Neukládat canonical prompt, baseline, skills ani provozní stav jako zdroj pravdy do Memory
- Pokud agent uloží Memory poznámku: reportovat ji jako pomocnou kontextovou poznámku, ne jako canonical stav
- Canonical zdroje mají vždy přednost: Gitea → Drive → DB cache → bootstrap (v tomto pořadí)

## Ntfy Runtime Rules

Ntfy posílat pouze přes MCP `send_notification` tool, pokud je skutečně dostupný jako callable tool.
Reportovat jeden z těchto stavů v run reportu:
  ntfy: sent
  ntfy: skipped_capability_missing
  ntfy: failed_403  (chybí NTFY_USER/NTFY_PASS v kontejneru — informovat operátora)
  ntfy: failed_unconfirmed
Ntfy failure je reporting degradation, ne automatický blocker datového běhu,
pokud DB zápisy a validace proběhly bezpečně.
