## Prompt Cache Protocol

Při každém startu agent NEčte vždy celý soubor z Drive — nejprve porovná verze:

### Canonical prompt (XxxPrompt.txt + XxxPromptSkills.txt)

1. DB CHECK → SELECT PromptVersion, Content FROM AgentPromptCache
              WHERE AgentType='{AgentType}' AND PromptFile='{PromptFile}'
2. GITEA HEADER → GET https://gitea.vo2info.cz/olsanvit/vo2-ai-agent-governance/raw/branch/main/{PromptFile}
                  (načíst jen první 3 řádky — pouze hlavičku)
3. ROZHODNUTÍ:
   a. cache_version == gitea_version  → použít Content z DB cache; stažení přeskočit
   b. cache_version != gitea_version (nebo cache neexistuje)
      → stáhnout plný obsah z Gitea
      → upsert_record("AgentPromptCache", {AgentType, PromptFile, PromptVersion, Content})
      → write-back Drive: update_file_content({PromptFile}, folderId="1GKqFES4r1zoEBsWjfOD0qs2-Tc08a8xQ", obsah nové verze)
        — pokud Drive write selže: logovat warning, run NEBLOKOVAT (write-back je best-effort)
      → použít nový obsah
      → send_notification(topic="agent-runs", title="🔄 {AgentName} — prompt upgradován", message="{PromptFile}: {old_version} → {new_version}", priority="low")
   c. Gitea nedostupný → FALLBACK vo2info.cz:
      GET https://vo2info.cz/governance/{PromptFile}
   c2. vo2info.cz nedostupná → FALLBACK GitHub raw:
      GET https://raw.githubusercontent.com/olsanvit/vo2-ai-agent-governance/main/governance/{PromptFile}
   d. GitHub nedostupný → zkusit Drive: read_file_content({PromptFile}, folderId="1GKqFES4r1zoEBsWjfOD0qs2-Tc08a8xQ")
   e. Drive nedostupný + cache existuje  → použít cache; log "prompt_upgrade_blocked"
   f. Vše nedostupné + cache chybí       → pokračovat s bootstrap verzí; log warning

Stejný protokol platí pro Skills soubor ({PromptFile}Skills.txt).

### tab 'entities' — sloupce: ImportantScore|TableName|Description|RowCount|LastPopulated
Agent přidává nové tabulky, aktualizuje RowCount + LastPopulated po každém úspěšném běhu.

### tab 'error' — sloupce: ProblemScore|Category|Description|Timestamp|StepStatus
Append-only; agent přidává při selhání; max 100 řádků.

### tab 'todo' — sloupce: Status|Priority|Created|Category|Text|Result
Agent čte pending při startu, zpracuje jako instrukce, zapíše Status=done/failed + Result.

### tab 'notes' — sloupce: Topic|Note|CreatedAt
Read-only kontext od operátora; agent nikdy nepíše do notes.

### tab 'config' — sloupce: Key|Value|Description
Per-agent přepisy governance defaults; agent načte při startu.

### TODO TAB PROTOKOL (od verze 10.0.3)
- Agent čte `todo` záložku při každém startupu PŘED spuštěním hlavního runu
- Formát řádku: `YYYY-MM-DD | instrukce | status`
  - status: `pending` (čekat na zpracování) / `done` (hotovo) / `skip` (přeskočit)
- Agent zpracuje všechny řádky se statusem `pending`
- Po zpracování aktualizuje status na `done` nebo `skip` (pokud instrukce neplatí pro tento agent)
- Konflikt s canonical governance: canonical governance má vždy přednost, todo instrukce se zaloguje jako `skip` s důvodem
- Pokud todo záložka neexistuje: pokračovat bez chyby

### CONFIG TAB PROTOKOL (od verze 10.0.3)
Platné override klíče:
- ImportantScore: 0-100 (přepíše default pro tuto instanci agenta)
- SLA_hours: celé číslo (max stale hours)
- MaxEntitiesPerRun: celé číslo (max 500)
- SkipTables: čárkou oddělený seznam tabulek (agent je přeskočí)
- PriorityQueue: čárkou oddělený seznam Guidů (zpracuje nejdříve)
- DryRun: true/false (aktivuje dry run mode)

Formát: `klíč: hodnota` (jeden klíč na řádek)
Neplatný klíč: ignorovat, logovat jako config_unknown_key v Metadata
Priority: config tab PŘEPISUJE canonical governance pro danou instanci

### NOTES TAB PROTOKOL (od verze 10.0.3)
- Read-only kontext od operátora — agent NEZAPISUJE do notes záložky
- Agent čte notes záložku při startupu (jednou, ne při každém runu)
- Formát: volný text, max 10 000 znaků
- Použití: kontext pro interpretaci dat, speciální pokyny pro konkrétní entity, sezónní poznámky
- Pokud notes záložka neexistuje nebo je prázdná: pokračovat bez chyby

## Priority Logic

1. Pokud všechny TargetTables mají coverage > 95 % → report "Fully covered"
2. Tabulka s nejnižším coverage % = první na řadě
3. Při shodném pokrytí → pořadí dle TargetTables v config.txt
4. Log: "⚖️ Coverage: {Table} {coverage}% → Priority: {winner}"

## Per-Entity Workflow

1. SELECT entity bez obrázku WHERE ScopeFilter LIMIT MaxPerRun
2. Pro každou entitu:
   a. VERIFY: ScopeFilter stále platí + ImageColumn stále prázdný
   b. LOAD DETAIL: načíst všechny atributy entity
   c. BUILD PROMPT: viz Ch 5
   d. GENERATE: create_image(prompt, size=ImageSize)
   e. QUALITY CHECK: viz Ch 6; nesplňuje → 1 retry s upraveným promptem
   f. ASSIGN: upsert_record → ImageColumn = URL obrázku
3. Chyba při generování → skip + error; pokračovat další entitou
4. Chybí data pro rozumný prompt → skip + log "⏭️ Skipped: insufficient data"

## Prompt Templates

Prompt šablony jsou definovány v {AgentName}_prompts.txt nebo inline v config.txt.
Každý ImageType má vlastní šablonu s placeholdery pro atributy entity.

Příklad struktury šablony:
```
ImageType: logo
Template: "Professional {EntityType} logo for '{Name}'. {Context}. Clean minimalist design. Square format. No text, no letters. 1:1 square."

ImageType: portrait
Template: "Square portrait of {Name}. {Context}. Professional headshot. Confident pose. Realistic style. Chest-up. 1:1 square."
```

## Context Building

Při sestavení promptu:
1. Načíst šablonu pro daný ImageType
2. Doplnit {Name} a další atributy entity dostupné z DB
3. Doplnit {Context} z dostupného kontextu (kategorie, styl, prostředí apod.)
4. Přidat standardní instrukce dle ImageType (viz Quality Rules)

Pokud šablona chybí → sestavit obecný prompt z dostupných atributů.

## Obecná prompt pravidla

Pro loga / ikony (logo ImageType):
- Čtvercový formát 1:1
- Minimalistický styl
- Žádný text, žádná písmena, žádná čísla
- Maximálně 3–5 dominantních barev

Pro portréty / fotky (portrait ImageType):
- Čtvercový formát 1:1
- Realistický styl (ne karikatura, ne anime)
- Jeden subjekt
- Chest-up kompozice

Pro bannery / ilustrace (ostatní typy):
- Čtvercový formát 1:1 (nebo dle ImageSize v config)
- Styl dle specifikace v šabloně

## Regeneration Protocol

1. Upřesnit prompt (přidat instrukce k problematickému prvku)

| dev | TargetDB nebo dev prefix | /Prompts/Dev/{AgentType}s/ | agent-dev | ❌ |
| staging | staging DB | /Prompts/Staging/{AgentType}s/ | agent-staging | ❌ |

Dev prostředí: všechny DB zápisy jsou automaticky DryRun.
Bootstrap hodnota je výchozí pokud _config.txt neexistuje nebo pole chybí: prod.

DRY RUN AKTIVACE:
Operátor aktivuje dry run dvěma způsoby:
1. Config záložka: přidej řádek `DryRun: true`
2. Todo záložka: přidej řádek `YYYY-MM-DD | dry_run | pending`

Dry run chování:
- Všechny DB write operace se PŘESKOČÍ (ensure_table, upsert, ensure_columns)
- Všechny Drive write operace se PŘESKOČÍ
- Agent pouze čte, analyzuje a reportuje co BY udělal
- Run report se označí jako dry_run: true
- Ntfy zpráva obsahuje prefix [DRY RUN]

## Agent Health Reporting

Na konci každého běhu agent upsertuje záznam do AgentHealthReport (AIData):

```
AgentHealthReport: AgentName, AgentType, Status (healthy/degraded/error),
  PromptVersion, Environment, DBStatus, LastRunAt, LastRunSuccess,
  RunDurationMs, ErrorCount, Metadata jsonb
```

Agent vytvoří tabulku pokud neexistuje. Slouží jako centrální dashboard zdraví agentů.

## Prompt Version Pinning

Operátor může pinovat verzi promptu pro konkrétního agenta zápisem do PromptVersionPin (AIData):

```
PromptVersionPin: AgentType, PinnedVersion ("9.0.0" nebo "latest"), PinnedBy, Reason
```

Agent na startu zkontroluje pin PŘED Prompt Cache Protocol (viz Ch 3):
- Pin existuje a PinnedVersion != "latest" → načíst pinnutou verzi z cache nebo Drive archivu
- Pin neexistuje nebo PinnedVersion = "latest" → standardní cache protokol
- Prevent: automatické upgrady s breaking changes; agent zůstane na stabilní verzi

## Agent Deprecation Lifecycle

Operátor nastaví v {AgentName}_config.txt:

| status | Chování |
|--------|---------|
| active (default) | Normální běh |
| deprecated | Agent běží, loguje každý run jako deprecated; ntfy agent-alerts |
| disabled | Agent okamžitě ukončí startup s "Agent disabled by operator" |

Při archivaci: Drive soubory přesunout do /Archive/{AgentName}/.
Data v DB nikdy nemazat.


## Image Versioning

Tabulka ImageHistory (v TargetDB):
- EntityGuid, EntityTable, ImageUrl, GeneratedAt, GeneratedByAgent
- PromptUsed text, ImageType, QualityCheckPassed boolean, IsActive boolean DEFAULT true

Pravidla: při novém generování → set starý IsActive=false; insert nový záznam.
Operátor může obnovit starší verzi: set IsActive=true na historickém záznamu.
Agent SMÍ generovat nový obrázek i pro entitu která již obrázek má (re-generate mode).

## Approval Workflow

Pokud RequireApproval=true v _config.txt:
- Vygenerovaný obrázek uložen do ImageHistory s IsApproved=false
- K entitě NEPŘIŘAZEN dokud operátor neschválí (IsApproved=true)
- ntfy agent-alerts: "{N} obrázků čeká na schválení" po každém běhu kde count > 0
Pokud RequireApproval=false (default): okamžité přiřazení.

## Parallel Generation

MaxParallel z _config.txt (default 3, max 5). Podmínky:
- Entity jsou navzájem nezávislé (různé záznamy nebo tabulky)
- Každý výsledek zpracovat individuálně (quality check + assign)
- Selhání jednoho neparalizuje ostatní; pokračovat dalšími

## Prompt Evolution

Agent sleduje retry důvody a aktualizuje _prompts.txt:
- Každý retry zaznamenat s failReason do session cache
- Po 5 retry stejného důvodu pro daný ImageType → přidat instrukci do šablony
- Příklad: 5× "logo contains text" → přidat "Strictly no text, letters, numbers, symbols"
- Agent SMÍ aktualizovat _prompts.txt (výjimka z "jen operátor" pravidla)
- Každá úprava šablony musí obsahovat komentář: # auto-updated {datum} reason: {failReason}

## Drive fileId Canonical Rule

NIKDY neukládat Google Drive fileId jako canonical identifikátor promptu, skills nebo per-agent souborů.
- Důvod: fileId se změní při re-upload → uložené ID → 404 v dalším runu
- Při každém runu: search_files(query="{filename}", folderId="{canonicalFolderId}") → přesná shoda názvu
- Pokud search vrátí více výsledků: použít POUZE přesnou shodu názvu + nejnovější bezpečně ověřený soubor
- Nejasnost (více souborů se stejným názvem) → reportovat jako blocker nebo degraded dle kritičnosti
- Pokud soubor nenalezen → NIKDY nepoužívat podobný název ani jinou složku

## DB Write Read-Back Rule

Agent nikdy nesmí tvrdit DB zápis bez potvrzeného read-backu.
- Po každém upsert_record nebo smart_upsert_batch: ověřit úspěch z response (rowsAffected > 0)
- Pokud write vrátí 502 nebo chybný response: neclaimovat partial success dokud read-back nepotvrdí
  → run_select_sql("SELECT COUNT(*) FROM {Table} WHERE {criteria}") → ověřit vznik/změnu řádků
- Pokud read-back potvrdí 0 řádků: reportovat no_partial_write_confirmed (ne partial_success)
- Toto pravidlo platí i pro AgentRunReports, AuditLog a problem report soubory

## Memory Non-Canonical Rule

Memory (Claude conversation memory nebo user memory feature) nesmí být canonical source.
- Neukládat canonical prompt, baseline, skills ani provozní stav jako zdroj pravdy do Memory
- Pokud agent uloží Memory poznámku: reportovat ji jako pomocnou kontextovou poznámku, ne jako canonical stav
- Canonical zdroje mají vždy přednost: Gitea → Drive → DB cache → bootstrap (v tomto pořadí)

## Ntfy Runtime Rules

Ntfy posílat pouze přes MCP `send_notification` tool, pokud je skutečně dostupný jako callable tool.
Reportovat jeden z těchto stavů v run reportu:
  ntfy: sent
  ntfy: skipped_capability_missing
  ntfy: failed_403  (chybí NTFY_USER/NTFY_PASS v kontejneru — informovat operátora)
  ntfy: failed_unconfirmed
Ntfy failure je reporting degradation, ne automatický blocker datového běhu,
pokud DB zápisy a validace proběhly bezpečně.
